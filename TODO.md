## Phase 2: аварийное резюме сессии для продолжения после /clear

**Статус:** идея (не начато). Phase 1 — детект + эксцепт мусора в ТГ — уже в проде.

### Цель
При cross-provider загрязнении контекста (ошибка 400 — чужой thinking без
подписи / `server_tool_use` с id не `srvtoolu_…` / смещённый `tool_result`) не
просто советовать `/clear`, а **сэкономить суть разговора**: отрезать мусорный
хвост, прогнать чистую голову через модель и выдать в ТГ короткое резюме —
чтобы после `/clear` скормить его и продолжить с нуля, без потери контекста.

### Что уже есть (строительные блоки)
- `core/sessions.py`: `transcript_path(session)` → `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`;
  `_scan_pollution(entries)` — чистая функция, находит маркеры загрязнения;
  `read_pollution_excerpt(session)` — читает хвост транскрипта.
- `core/turn.py`: релей `_error_relay_loop` уже ловит `protocol`-400 и шлёт эксцепт.
- modelpipe-прокси на `127.0.0.1:8787` — туда можно слать свежий Anthropic-запрос
  на резюме (роут `claude-*` напрямую, или дешёвый `deepseek-*` / `glm-*`).
- `core.notice(session, text)` — доставка длинного текста во все адаптеры.
- **Проверенный факт:** Claude Code персистит `signature` у thinking-блоков
  (7770/8702 подписаны на момент проверки) → неподписанный thinking = реально
  чужой бэкенд, детектор не ложноположит.

### Наметка потока
1. На `protocol`-400 (или по кнопке — см. развилку) прочитать весь транскрипт.
2. Найти границу загрязнения (`_scan_pollution` уже умеет;расширить — отдавать индекс,
   а не только эксцепт).
3. Отрезать хвост от первого загрязнённого блока → чистая голова.
4. Сложить чистую голову в один user-запрос: «сожми состояние этой сессии в
   резюме для старта с нуля — что делали, где остановились, что следующее».
5. POST `http://127.0.0.1:8787/v1/messages` с этим запросом (свежий call,
   модель не отравлена).
6. Ответ-резюме → в ТГ: «после /clear скорми это: <резюме>».

### Развилки, которые надо решить (не угадывать)
1. **Триггер.** Автоматически при каждом 400, или кнопкой в сообщении Phase 1
   (чтобы не тратить токены/латентность зря)? Склоняюсь к кнопке.
2. **Где резать.** По первому загрязнённому блоку (потеряем свежие чистые ходы
   после него) или вырезать только мусорные пары tool_use/tool_result (точнее,
   но сложнее и ломает парность). Для v1 — по первому блоку, проще и надёжнее.
3. **Модель для резюме.** Дешёвый `deepseek-v4-pro` (1M окно) или `glm`? claude-*
   дороже, но точнее. По умолчанию — deepseek.
4. **Длиной резюме управлять.** Чанковать чистую голову, если не лезет в окно
   модели-резюмёра (у deepseek 1M — обычно не проблема).

### Риски
- Сам call резюме идёт через modelpipe → кормить ему **только очищенный** хвост,
  иначе упадёт на том же мусоре (110% обязательно).
- Если modelpipe лежит / 429 — резюме падает, но Phase 1 (эксцепт + /clear)
  должен остаться рабочим. Резюме = soft-enhancement, не блокер.
- Транскрипт может быть большим — читать целиком каждый 400 дороговато; для v1
  терпимо (событие редкое), потом можно читать хвост чанками с конца.

### Минимальный первый шаг (если хватит)
Кнопка «📋 Резюме для /clear» в сообщении Phase 1 → по клику один modelpipe-call
на дешёвой модели с обрезанным по первому мусорному блоку хвостом → ответ в ТГ.
