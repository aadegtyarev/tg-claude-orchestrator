# agent-vm: варианты доработки под наши нужды

Исследование 2026-07-22. Вопрос: как получить от agent-vm то, чего не хватает
(см. «Проблема» ниже) — перенастроить, расширить, обернуть, форкнуть или
написать своё. Все утверждения ниже — из живых экспериментов на этой машине
(agent-vm 0.1.25), не из предположений.

## Проблема (кратко)

bwrap — прозрачная граница: env/PATH/$HOME/сеть процесса claude — наши.
agent-vm — непрозрачная: креды, профиль, сеть и окружение гостя принадлежат
ему, точек расширения не документировано. Не решено: выбор учётки Claude
(разные `CLAUDE_CONFIG_DIR`), кошелёк в VM, транскрипты/stats, свой прокси
без своего токена.

## Ключевые находки экспериментов (меняют картину)

### 1. State-каталог — это НЕ чёрный ящик, а хостовая папка

`/root/.claude` гостя — симлинк на `/agent-vm-state/claude`, а `/agent-vm-state`
— bind-mount **хостового** каталога:

```
~/.local/state/agent-vm/<sha256(cwd)[:12]>/
  claude/            ← гостевой /root/.claude ЦЕЛИКОМ (settings.json, projects/…)
  claude.json        ← гостевой /root/.claude.json
  gitconfig, codex, opencode…
~/.local/state/agent-vm/<hash>.secrets/
  anthropic          ← реальный OAuth-токен (0600, вне видимости гостя)
```

Проверено живьём:
- **хэш воспроизводим**: `sha256(cwd_путь)[:12]` (сверено: `/tmp` → `e9671acd2448`);
- **файл, записанный с хоста в `state/claude/`, виден в госте** (`MARKER=[маркер-с-хоста]`);
- **транскрипты гостя читаются с хоста**: `state/<hash>/claude/projects/<encoded-cwd>/*.jsonl`
  — то есть `/stats`, детект загрязнения и Phase-2-резюме чинятся БЕЗ каких-либо
  доработок agent-vm, надо только вычислить путь;
- в гостевые креды кладётся **плейсхолдер** (`msb-anthropi…`), реальный токен
  остаётся в `.secrets` на хосте — подмена на проводе (как задокументировано).

⚠️ Запись в state-каталог — недокументированное поведение: после моих записей
проект `/tmp` перестал грузиться (`mount workspace: Operation not permitted`;
на чистом проекте всё работает — видимо, задел оверлей-учёт msb). Пользоваться
можно, но аккуратно (класть файлы, не трогать структуру) и с готовностью, что
апстрим это поменяет.

### 2. Учётка выбирается через $HOME процесса agent-vm

`CLAUDE_CONFIG_DIR` agent-vm **игнорирует** (замерено: креды сеются из
`~/.claude` даже при переменной на пустой каталог). Но `~` — это `$HOME`
процесса agent-vm: с подменённым `$HOME` креды взялись из него (гость получил
`.credentials.json` из симлинка в fake-HOME; с пустым fake-HOME пропала и
gh-идентичность). Костыли: надо сохранять `XDG_STATE_HOME` (иначе уезжает кэш
образа) и следить за длиной пути (лимит Unix-сокета 108 байт).
Секреты **пересеиваются при каждом запуске** (mtime `.secrets/anthropic`
меняется) — подкладывать свои в `.secrets` бесполезно, источник — `$HOME/.claude`.

### 3. env гостя настраивается внутри, но флага нет

`AGENT_VM_DEBUG_CONFIG=1` дампит полный SandboxConfig JSON — там есть массив
`env` (CODEX_HOME, PATH, IS_SANDBOX…). То есть механизм задать env гостю в
microsandbox ЕСТЬ, agent-vm просто не даёт его снаружи. Апстрим-патч «флаг
`-e KEY=VAL`» тривиален.

### 4. Архитектура пакета

npm-пакет — тонкий JS-лончер; вся логика в **Rust-бинарях** платформенного
сабпакета (`agent-vm` + патченный `msb` + libkrunfw), исходники открыты
(github.com/wirenboard/agent-vm, microsandbox — их же форк). Форк = своя
сборочная цепочка нативных бинарей и догонялка за активным апстримом
(ежечасные пересборки образа-шаблона).

## Варианты

### A. Использовать найденные точки расширения (без изменений agent-vm)

Что закрывает уже сейчас:
- **транскрипты/stats** — читать `state/<hash>/claude/projects/` с хоста;
  hash = `sha256(effective_cwd)[:12]`. Малый код в `transcript_path()`.
- **профиль (плагины/агенты/настройки)** — класть файлы в `state/<hash>/claude/`
  при провижне сессии (осторожно, см. предупреждение) ИЛИ передавать флагами
  `--plugin-dir`/`--agents`/`--settings` с `--mount` в нейтральный путь
  (проверено: работает).
- **учётка per-инсталляция или per-cwd** — запуск agent-vm с нужным `$HOME`
  (+`XDG_STATE_HOME`). Ограничение: одна учётка на ЗАПУСК CLI, т.е. на сессию —
  выбирается при старте сессии; этого хватает для «разные проекты — разные
  учётки».

Плюсы: ноль зависимостей от апстрима, всё проверено живьём, малые объёмы.
Минусы: опора на недокументированное (state-layout, $HOME-сидинг) — нужен
smoke-тест на обновление agent-vm; запись в state требует аккуратности.

### B. Свой кред-прокси на хосте (наш слой вместо/рядом с их прокси)

Сервис на хосте: per-session непрозрачный токен → реальные креды нужной
учётки, подстановка на проводе + OAuth-refresh (учётки — claude.ai OAuth,
проверено по структуре `.credentials.json`). Сессия получает
`ANTHROPIC_BASE_URL` нашего прокси (сетевой путь ПРОВЕРЕН: LAN-IP хоста +
`--allow-egress` + свой токен → запросы дошли).

Закрывает: **учётки per-session** (сильнее, чем $HOME — можно менять на лету),
**кошелёк для HTTP-секретов** (плейсхолдер в госте, подстановка на хосте — та
самая отложенная «TLS-прокси» из secrets-wallet.md, изоляция НЕ ослабляется).
Не закрывает: ssh/scp (не HTTP), профиль, транскрипты (это вариант A).
Минусы: самая большая работа (OAuth-refresh, ротация, безопасность нового
сервиса); отдельный проект-соблазн — НЕ тащить в modelpipe (решение оператора).

### C. Свой OCI-образ (`AGENT_VM_IMAGE`)

Запечь в образ профиль/пакеты. Плюсы: официальная ручка, конфликтов нет.
Минусы: не решает креды (их сеет agent-vm), поддержка своего образа против
ежечасного апстримного, риск запечь секрет. Годится как дополнение к A для
тяжёлых кастомизаций гостя (python-пакеты, инструменты).

### D. Апстрим-PR в wirenboard/agent-vm

Кандидаты (все по находкам тривиальны их коду): `-e KEY=VAL` (env уже есть в
SandboxConfig), `--claude-profile <dir>` (источник сидинга кред вместо
`$HOME/.claude`), разрешить `--mount` в `/root/.claude/<подкаталог>`.
Плюсы: чисто, навсегда, полезно всем пользователям agent-vm. Минусы: сроки
зависят от мейнтейнеров; до принятия жить на варианте A.

### E. Форк agent-vm

Полный контроль. Минусы серьёзные: Rust + сборка нативных бинарей (patched
msb, libkrunfw), вечная догонялка за активным апстримом ради 2–3 флагов.
Оправдан ТОЛЬКО если D отклонят, а A перестанет работать.

### F. Свой лончер поверх msb (микросэндбокса)

agent-vm — обёртка над патченным `msb`; SandboxConfig мы видим целиком
(`AGENT_VM_DEBUG_CONFIG`), можно собирать свой и звать msb напрямую — свои
env/mounts/креды без ограничений. Минусы: теряем их кред-прокси, gh-прокси,
egress-политику и OAuth-refresh — всё это пришлось бы переписать (по сути
вариант B + куски E). Это «свой agent-vm»: дороже форка по итогу.

## Рекомендация

Порядок по цене/ценности:
1. **A сейчас** — транскрипты/stats с хоста (чинит /stats под VM), профиль
   флагами, учётка через $HOME per-session. Всё проверено, малые срезы.
2. **D параллельно** — PR-ы в апстрим (`-e`, `--claude-profile`): убирают
   недокументированность A.
3. **B как следующий крупный шаг** — если понадобятся учётки «на лету» и
   кошелёк-подстановка HTTP-секретов в VM.
4. E/F — держать как запасной план, не начинать без отказа апстрима.
