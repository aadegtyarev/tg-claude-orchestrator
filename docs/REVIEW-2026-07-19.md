## Модель угроз (по оператору — читать первым)

**Цель системы — страховки от СЛУЧАЙНЫХ глупостей модели и утечки токена наружу, НЕ глубокая изоляция от враждебной/заинъекченной модели.**

Следствия для приоритизации:
- Сеть, доступ к другим машинам, запуск Docker — **нужные возможности**, не дыры. Модель осознанно выполняет код на хосте (через кошелёк) — это by design.
- Находки вида «модель НАМЕРЕННО ломает изоляцию / выполняет RCE на хосте» — **понижены** (это её штатная работа, а не эксплойт).
- В цель остаются: **не выкинуть токен наружу случайно**, не самоаппрувнуть/не перепутать сессии по случайности, не сломаться на кривом вводе, веб-origin не должен исполнять чужой HTML.

Каждая находка ниже помечена актуальностью под эту рамку.

## 1. Безопасность (под калиброванной моделью угроз)

### В цель (страховки/утечки) — оставить high/medium

- ✅ **[high] web/adapter.py:568-582 `h_file` отдаёт файлы INLINE (без `Content-Disposition`) → исполнение HTML на origin веба.**
  Модель пишет `report.html` c `<script>` в проект, шлёт `send_file_to_user` → пользователь кликает `/file?path=...` → браузер рендерит на том же origin → скрипт делает `fetch('/api/.../bash')` с cookie → захват API (bash, approve permission, чтение всех сессий). `h_log` (:594) уже ставит `attachment` — `h_file` нет.
  **Фикс:** `Content-Disposition: attachment` + `X-Content-Type-Options: nosniff` в `h_file`. (Актуально даже под «случайными глупостями» — это про веб-origin, не про изоляцию модели.)
- **[medium] wallet/module.py:239-252 `_redact` — literal-only, подаётся барьером.**
  Заменяет только точные вхождения; любая трансформация (base64/hex/reverse/запись в файл) проходит. Это про «случайно эхнуть токен» — гигиена, не барьер.
  **Фикс:** в доках/комментах явно «редакция стопит только случайный echo, не защита»; реальный контроль — host-passthrough (нет значения) + не давать шелл в commands.
- ✅ **[medium] wallet/module.py:277-287 + sessions.py:418 — wallet под `SANDBOX=off`/`agent-vm` не страхует.**
  Под `off` `secrets.toml` читается моделью напрямую (нет песочницы), инвариант «секрет не в пространстве модели» пуст; deny/autoMode пишутся только при `bwrap`. **Решение оператора: НЕ hard-error** (кошелёк может работать и без песочницы), а **громкое предупреждение в логах при старте** + документация: «кошелёк без `SANDBOX=bwrap` НЕ страхует от утечки — использовать ТОЛЬКО вместе с песочницей». Предупреждение уже есть (module.py:277-287) — усилить формулировку («не страхует / используй вместе»), продублировать в README/secrets-wallet.md. В цель: оператор осознаёт, что связка обязательна.
- ✅ **[medium] app.py:722-729 — временный INFO-лог полного permission-payload** (превью команд с потенц. секретами в общий лог). **Фикс:** убрать (это остаток отладки причины эскалации).
- ✅ **[medium] sessions.py:63-67,93-104 `_pty_driver` авто-доверяет linked-папкам** (`trustthisfolder`/`bypasspermissions` авто-Yes). `/new <path>` на чужой репо грузит его project-хуки без согласия (MCP-авто-апрув корректно снят для linked — sessions.py:406, но hooks нет). **Решено (warn+doc, не hard-block — как wallet-без-sandbox):** `_warn_project_trust` ругается в лог при линковке папки с `.claude/settings[.local].json`-хуками или `.mcp.json` — «линкуй только доверенные репо». Полный запрет отверг: startup завис бы без TUI-подтверждения, а папку оператор выбирает сам (вне рамки враждебного репо).
- ✅ **[medium] core/hookscript.py + sessions.py:555 — ORCH_TOKEN читается моделью** (session_dir примонтирован RW, тот же uid — модель может `cat` 0600-скрипт). Под нашей рамкой это не «враг ломает гейт», а **случайный самоаппрув / кросс-сессионная путаница**: с токеном можно POST `/permission behavior=allow`, `/reply`, `/notify`. Комментарии hookscript.py:4-8 обещали секретность — по факту 0600 стопит только чужих unix-юзеров. **Решено (принять в модели угроз):** коммент hookscript.py исправлен — 0600 прячет токен от ДРУГИХ unix-юзеров, но не от модели; ORCH_TOKEN считается известным модели, под рамкой «страховки, не изоляция от враждебной модели» это принято. Полная развязка токенов (capability report-only vs verdict/notify/reply вне песочницы) — опционально, вынесена как возможное усиление, но не требуется под текущей рамкой.
