## Модель угроз (читать первым — из ревью 2026-07-19, подтверждена)

**Цель системы — страховка от СЛУЧАЙНЫХ ошибок модели и утечки токена НАРУЖУ, НЕ
глубокая изоляция от враждебной/заинъекченной модели.** Сеть, доступ к другим
машинам, запуск Docker, выполнение кода на хосте через кошелёк — штатные
возможности, не дыры. Находки «модель НАМЕРЕННО ломает изоляцию / RCE на хосте»
понижены/отброшены. В цели остаются: не выкинуть секрет наружу случайно, не
самоаппрувнуть/не перепутать сессии, не сломаться на кривом вводе, веб-origin не
исполняет чужой HTML.

- ✅ **[warning] orchestrator/modules/wallet/policy.py:241-244 — `secrets.toml.tmp`
  пишется с широкими правами до `chmod` (значения секретов в окне гонки).** СДЕЛАНО:
  `_write` создаёт temp через `os.open(..., O_CREAT|O_TRUNC, 0o600)` (как `_provision`).
  `_write` при каждой правке `/wallet` перезаписывает весь документ секретов

  модели угроз (перекос прав на файле секретов). Эксплойт требует второго
  локального пользователя в узком окне.
  **Фикс:** создавать temp сразу с 0600, как в `_provision`:
  `fd = os.open(tmp, os.O_WRONLY|os.O_CREAT|os.O_TRUNC, 0o600)` → запись → `os.replace`.

## SUGGESTIONS

- ⬜ **[suggestion] orchestrator/core/bubble.py:377 — полный скан `/proc` на каждый
  flush бабла (~1.5 с) на каждую активную сессию.**
  `_flush` → `_unblock_available` → `unblock_action` → `manager.has_children` →
  `proctree.proc_tree_signals` делает `os.listdir("/proc")` + читает
  `/proc/<pid>/stat` для КАЖДОГО процесса хоста. Кроме случая last-tool=`TaskOutput`,
  это происходит на каждый flush (~раз в `EDIT_INTERVAL`=1.5 с) на каждую сессию с
  активным ходом — при нескольких сессиях это O(все-процессы-хоста) несколько раз в
  секунду ради решения, показать ли кнопку ⏭. `is_busy` (watchdog, 15 с) считает то
  же дерево независимо.
  **Фикс:** кэшировать результат `proc_tree_signals` кратко (memoize по pid ~1 с)
  или переиспользовать значение watchdog'а, чтобы per-flush-проверка кнопки не
  запускала отдельный скан `/proc`.

- ⬜ **[suggestion] orchestrator/core/bubble.py:399-407 — `_flush` пере-планирует
  себя каждые 1.5 с без backoff, если все транспорты не доставляют.**
  `sent_text` обновляется только при `delivered=True`. Если каждый транспорт
  устойчиво бросает (Telegram отвергает: 400 «can't parse entities», сообщение стало

  1.5 с бесконечно. Не-backed-off ретрай против заведомо неуспешного запроса, пока
  бабл открыт (завершается только на close/teardown).
  **Фикс:** считать подряд неудачные flush и прекращать пере-планирование (или
  экспоненциальный backoff) после N неудач; возобновлять на следующем реальном
  изменении состояния.

## Архитектура и дизайн (forward-looking, не блокирует)

- **app.py / OrchestratorCore: эфемерное per-session-состояние размазано, teardown
  чинит его вручную.**
  `OrchestratorCore` (1463 строки) хорошо разбит и слоистость чистая (её сохраняем).
  Но класс стал домом для нескольких самодостаточных подсистем со своим per-session
  состоянием: hook→bubble трекинг (`_last_tool`, `_agent_types`, `_agent_spawns`),
  снапшоты фоновых задач (`Session.background_tasks/session_crons/bg_seen`), релей
  permission (`_pending_perms`, `_local_perms`). Признак — `_teardown_runtime`
  (app.py:359-387): один метод обязан помнить про `pop` каждого дикта; комментарий
  автора там фиксирует, что последовательность «УЖЕ разъехалась» (idle чистил
  `_last_tool`, остальные — нет) и её свели руками. Не баг; readability/
  maintainability, важно при следующем добавлении per-session-состояния.
  - *Вариант A — оставить как есть* (effort: none). Плюсы: ноль риска, код работает и
    отлично прокомментирован, вся логика в одном файле. Минусы: app.py растёт, каждая
    новая единица состояния снова открывает риск дрейфа, три подсистемы сцеплены с
    плумбингом core.
  - *Вариант B — вынести hook→bubble трекер в свой класс с собственным состоянием и
    `forget(name)`* (effort: small–medium). Плюсы: ~250 строк уходят из app.py,
    `_teardown_runtime` зовёт один `tracker.forget(name)` вместо тыканья в три дикта
    (убивает класс дрейф-багов), шов естественный. Минусы: +1 файл/индирекция;
    `unblock_action` читает `_last_tool` → трекеру нужен `last_tool(name)`.
  - *Вариант C — единый per-session `SessionRuntime` для ВСЕГО эфемерного состояния*
    (effort: medium–large). Плюсы: одно место на сессию, teardown = «drop runtime».
    Минусы: трогает аккуратный рабочий lifecycle+permission код во многих местах,
    у permission-futures и Session-полей разные времена жизни — смешение может
    размыть; ближе всего к широкому редизайну, который оператор отклонял.
  - **Рекомендация:** Вариант B — окупается (убирает задокументированный автором риск
    дрейфа), это механический вынос, а не редизайн, ложится в «сохранить channels,
    предпочитать малые шаги». C — полнее, но не стоит риска на работающем коде;
    всплывать, только если планируется много нового per-session-состояния. A вполне
    защитим при текущей прокомментированности.

## Что проверено и признано ОК (не чинить)

- **Слоистость** adapters → Transport → core → sessions → runners соблюдена везде:
  адаптеры не импортируют `runners`, `core/*` не тянет `aiogram`/`aiohttp.web`,
  standalone env-чтения `channel_server.py` документированы как намеренные.
- **Обработка ошибок** единообразна: `SessionError` → публичный `UserError` на
  границе core; адаптеры ловят `UserError` для юзера и `except Exception` +
  `logger.exception` для прочего. User-facing текст идёт через `core/texts.py`.
- **Документация** синхронна: README / .env.example / config.py / wallet /
  docs/secrets-wallet.md / agent-vm-integration.md / messaging-connector.md /
  CHANGELOG — кросс-проверены, расхождений нет.
- **Острые места стабильности** закрыты осознанно: fire-and-forget задачи,
  утечки fd/PTY на error-путях, TOCTOU портов, first-answer-wins на permission,
  гонки freeze/close бабла.
