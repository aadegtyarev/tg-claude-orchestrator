# Форк agent-vm: флаг `--egress-proxy` (набросок под upstream-PR)

Статус: план (2026-07-22). Задание — добавить в agent-vm ОДИН флаг, позволяющий
направить egress гостя на внешний прокси оператора, БЕЗ регрессий без флага,
так чтобы это можно было отправить PR в upstream. Не хардкод под наш кошелёк.

Контекст и замеры — `ARCHITECTURE-claude-box.md` (F10) и `DECISIONS-claude-box.md`
(Р3). Лицензия agent-vm — MIT.

## Зачем

Под agent-vm весь egress гостя прозрачно MITM-ится нативным прокси microsandbox
(свой CA, инъекция только github/anthropic, не конфигурируется). Наш кошелёк не
может вставить свою инъекцию произвольных секретов. Флаг «направь egress на мой
прокси» решает это и полезен любому, кому нужен корпоративный/свой egress-прокси
(общий кейс — примут в upstream).

## Контракт флага (предложение)

```
agent-vm claude --egress-proxy <URL> [--egress-ca <PEM>] [прочее] -- <args>
  --egress-proxy   куда направить egress гостя (напр. http://192.168.1.44:9000).
                   Не задан → поведение как СЕЙЧАС (дефолтный внутренний прокси).
  --egress-ca      PEM CA внешнего прокси; ставится в trust-store гостя, чтобы
                   гость доверял MITM-сертификату внешнего прокси. Не задан при
                   заданном --egress-proxy → внешний прокси считается не-MITM
                   (plain-passthrough) ЛИБО его CA уже доверен образом.
  env-алиасы: AGENT_VM_EGRESS_PROXY / AGENT_VM_EGRESS_CA (как у прочих флагов).
```

Семантика: при заданном `--egress-proxy` внутренний перехват гостя
перенаправляется на указанный внешний прокси (либо вместо внутреннего, либо
внутренний туннелирует upstream к нему — выбрать при реализации так, чтобы
инъекция github/anthropic либо отключалась, либо делегировалась внешнему; для
нашего кейса Vault берёт на себя ВСЁ, включая Anthropic — тогда чище «вместо»).

## Точка изменения

`vendor/microsandbox/crates/network/lib/http_proxy.rs` — там задаётся адрес,
на который заворачивается egress гостя, и CA. Флаг пробрасывается из CLI
agent-vm в SandboxConfig → в network-конфиг микросэндбокса.

## Требования приёмки (обязательны)

1. **Без флага — ноль изменений.** `agent-vm claude` без `--egress-proxy`
   работает 1:1 как upstream (тот же внутренний прокси, gh/anthropic-инъекция,
   egress-политика). Тест: снять дифф поведения на дефолтном сценарии.
2. **С флагом — egress идёт на внешний прокси**, гость доверяет его CA (если
   задан), инъекция внутреннего прокси не мешает (отключена/делегирована).
   Тест: наш Vault-прокси на LAN-IP видит расшифрованный запрос гостя и
   подставляет секрет (повторить спайк, но теперь наш прокси НЕ обойдён).
3. **Egress-политика (`--allow-egress`/public_only) сохраняется** — флаг меняет
   ТОЧКУ прокси, а не открывает сеть.
4. Документация флага в `--help` и README; env-алиас.

## Разведка ДО оценки сроков (не пропускать)

- Собирается ли agent-vm из исходников на этой машине? Toolchain: Rust-версия,
  сборка нативных `msb`/libkrunfw — вендорятся готовыми или собираются? (Это
  определяет, форк «пара дней» или «недели».)
- Как CLI-флаги пробрасываются в SandboxConfig (есть ли уже похожий сквозной
  флаг — скопировать паттерн).
- Есть ли у microsandbox УЖЕ generic-режим внешнего прокси (тогда флаг —
  тонкая обёртка, а не новая логика).

## Как это встаёт в наш проект

- Vault поднимает egress-прокси (тот же MITM-движок, что и для bwrap, §4.2) на
  хосте, отдаёт лончеру `url`+`ca`.
- `AgentVmRunner.wrap` (наш код) добавляет `--egress-proxy <url> --egress-ca <ca>`
  когда движок agent-vm И кошелёк активен.
- Установка: пока PR не влит — в инсталляции наш форк agent-vm; влит — апстрим.
  Кошелёк не зависит от факта форка (флаг либо есть, либо игнор).
- Инвариант: без кошелька/флага agent-vm ведёт себя как апстрим (Р0/«выключено =
  не существует»).
