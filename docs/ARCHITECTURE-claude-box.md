## 1. Цель

Сегодня проект — монолитный «оркестратор»: Telegram/Web-интерфейс, внутри
которого вросли запуск Claude Code под PTY, изоляция (bwrap | agent-vm | off) и
кошелёк секретов. Цель редизайна — разделить это на слои так, чтобы:

1. **Изолированный запуск claude с профилем и секретами работал БЕЗ
   оркестратора** — как самостоятельный CLI (`claude-box`), которым пользуется
   человек за терминалом или скрипт CI.
2. **Оркестратор стал надстройкой** над этим CLI: сессии, Telegram/Web, бабл,
   resume — UX поверх, а не владелец механики.
3. **Кошелёк стал автономным сервисом (Vault)** с плагинами-коннекторами под
   конкретные сервисы (Google Docs, Jenkins, MediaWiki, Discourse, …) и работал
   одинаково в обоих движках изоляции.

Реальные задачи оператора, под которые это строится: дать модели доступ к
MediaWiki / Jenkins / Discourse / Google Docs **со скоупом** (например «только
эти документы/папки»), запускать разные задачи под разными учётками Claude,
переключать движок изоляции одним флагом.

## 2. Слои

```
Слой 3  ОРКЕСТРАТОР   сессии, Telegram/Web, статус-бабл, resume, permission-relay
                       = клиент Слоя 2. Про креды/секреты НЕ знает.
        ────────────────────────────────────────────────────────────
Слой 2  LAUNCHER      CLI `claude-box`: «запусти claude в движке Y с профилем
                       и скоупом Z в каталоге C». Экспонирует PTY.
        ────────────────────────────────────────────────────────────
Слой 1  VAULT         демон: секреты, профили-указатели, policy, перехват
                       (bwrap: MITM+shims; agent-vm: реверс-эндпоинт, после
                       форка — MITM, Р3), аудит, плагины-коннекторы.
        ────────────────────────────────────────────────────────────
Слой 0  ENGINE        bwrap | agent-vm | off. УЖЕ ЕСТЬ: orchestrator/runners/
                       (Protocol Runner: preflight/wrap/unique_cwd/supports_prefix).
```

Ключевой примитив: **лончер экспонирует PTY с claude; у него два фронтенда** —
человек в терминале (standalone) и оркестратор (программно, много лончеров).
Один примитив, два потребителя.

Прообраз всей формы — сам agent-vm: он и есть Слои 0–2 для узкого случая
(Anthropic-креды + gh). Мы обобщаем Слой 1 на произвольные секреты и делаем
Слой 2 движко-агностиком.

Всё ниже установлено экспериментами на этой машине (agent-vm 0.1.25). Это
ограничения среды, на которых стоит дизайн:

## 4. Слой 1: Vault

### 4.1. Форма

Автономный демон на хосте + CLI (`vault …`) + конфиг. Клиенты (launcher,
оркестратор, CLI) ходят по локальному HTTP с per-client токеном (как сейчас
демон кошелька: 127.0.0.1 + bearer). Хранение policy/секретов — TOML-файл
(эволюция сегодняшнего `secrets.toml`, 0600), значения секретов — в нём же или
host-passthrough (кред-сторы хоста), как сейчас.

Vault **не знает про Telegram/оркестратор**. Для «спроса» (см. 4.6) он лишь
экспонирует хук; кто и как спрашивает — забота клиента сверху.

**Упаковка (решено):** монорепо, как сейчас (проект без build-system,
clone-install — это стиль репозитория, не меняем). `vault/` и `box/` —
подпакеты БЕЗ зависимостей оркестратора (нет `aiogram`; свой минимум). «Работает
без оркестратора» = демон Vault и `claude-box` запускаются и функционируют без
Telegram-бота и его процесса, из того же клона. Отдельные pip-артефакты/репы —
НЕ сейчас (не плодить инфраструктуру, которой в проекте нет); если позже
понадобится настоящая изоляция дистрибуции — отдельная задача.

### 4.2. Транспорты секрета — механизм РАЗНЫЙ по движкам (проверено спайком)

Механизм перехвата НЕ единый: bwrap и agent-vm устроены по-разному, и спайк
(MITM-прокси + самоподписанный CA, `curl` изнутри на реальный URL) это доказал.

**bwrap — прозрачный перехват, модель ходит на РЕАЛЬНЫЙ URL:**

| Как модель обращается | Механизм | Что видим для скоупа |
|---|---|---|
| именованный тул (`curl`/`git`/`ssh`/`scp`) | **exec-шим**: команда перехвачена в PATH, исполнена на хосте с кредом, TLS не трогается (сегодняшний кошелёк, расширить на HTTP-тулы) | командная строка |
| произвольный клиент (`python requests`, SDK) | **MITM-прокси** Vault: `HTTP_PROXY`+наш CA в trust-store; **проверено живьём — впрыснутый заголовок долетел** | расшифрованный запрос |
| клиент с `base_url`/endpoint | **конфиг-редирект** без MITM, самое надёжное | запрос к нашему эндпоинту |
| ssh по ключу | **ssh-agent forwarding**: ключ в агенте хоста, проброшен только сокет | целевой хост |

**agent-vm — ДВА режима:**

- **С форком (`--egress-proxy`, Р3 / `FORK-agent-vm-egress-proxy.md`) — целевой:**
  egress гостя заворачивается на наш Vault-MITM → под agent-vm перехват ПОЛНЫЙ,
  как под bwrap (строка «произвольный клиент → MITM-прокси» из таблицы bwrap
  выше работает и здесь). Это и есть паритет движков.
- **Без форка — обход (текущее, до влития/сборки форка):** наш прозрачный MITM
  НЕВОЗМОЖЕН (F10) — agent-vm сам MITM-ит весь egress своим CA, наш прокси
  обойдён. Тогда секреты доставляются только так:

| Как | Механизм | Ограничение |
|---|---|---|
| клиент с `base_url`/endpoint | **конфиг-редирект** на наш хост (base_url через `env`-блок, F1) — проверено (PR #48, Anthropic) | нужен конфиг-knob у клиента |
| явный сервис-эндпоинт | модель ходит на `http://<LAN-IP>/svc/<имя>/…` (plain-HTTP к нашему хосту, `--allow-egress`; наш реверс-прокси авторизует наверх) — agent-vm plain-HTTP к разрешённому хосту не переМITMивает | модель должна ЗНАТЬ адрес (из `VAULT-ACCESS.md`, §4.4) |
| произвольный `curl https://realsvc` с нашим секретом | **НЕ перехватить** (F10 — их MITM владеет) | такие задачи — под bwrap |
| Anthropic / gh | ведёт кред-прокси самого agent-vm | их креды, не наши |
| ssh по ключу | TCP-бастион Vault (LAN-IP) или мост сокета | не-HTTP+VM разрыв |

**Следствие для оператора (честно):** «дать модели произвольный сервис-секрет,
модель ходит на любой реальный URL» — под **bwrap** сразу; под **agent-vm** —
после форка (`--egress-proxy`), а ДО форка только через конфиг-редирект/явный
Vault-эндпоинт (модель направлена на наш адрес). Это не ограничение нашего кода,
а того, что до форка egress-перехватом гостя владеет agent-vm (F10). SSH-ключ
хоста: под bwrap — ssh-agent forwarding (ключ в машину не входит); под agent-vm
— бастион (форк это не меняет: SSH не-HTTP).

⚠️ **CA Vault в trust-store (для bwrap-MITM):** пишем в `/etc/ssl/certs` песочницы
+ `SSL_CERT_FILE`/`REQUESTS_CA_BUNDLE` (владеем ФС) — проверено спайком.

### 4.3. Атрибуция: чей запрос и какой скоуп

Один Vault обслуживает много сессий/VM. Привязка входящего запроса к
сессии/скоупу — **по выделенному на сессию listen-порту прокси/шима** (Vault
аллоцирует при `launch`, освобождает при завершении; пул портов, как у
channel-серверов). Source-bound: каждая сессия сконфигурирована на СВОЙ порт (bwrap:
`HTTP_PROXY`-порт MITM / сокет шима; agent-vm: порт реверс-эндпоинта или
base_url), поэтому Vault знает, чей запрос, без токена в самом запросе. Конфиг
лончер ставит: bwrap — env процесса + CA (§4.2); VM — `env`-блок настроек (F1).

### 4.4. Что получает модель (тиры) и как узнаёт скоуп

- **Дефолт (перехват прозрачный):** значение секрета в машину НЕ входит — прокси/
  шим подставляет его между машиной и сервисом. Прямой запрос модели к сервису
  без перехвата (клиент мимо MITM/шима, или до форка — произвольный HTTPS под VM)
  → 401 без кред («путь открыт, но пуст»), egress не ограничиваем (Р7).
- **Тир 3 (реальное значение) — escape hatch:** только МИНТ короткоживущего/
  скоупнутого под-токена (Jenkins API-token, GitLab PAT, OAuth exchange),
  выдача транспортом C. Мастер-секрет — лишь по ЯВНОМУ флагу policy
  `allow_master = true` (ОТДЕЛЬНОЕ поле, НЕ перегружать `confirm` — тот про
  подтверждение каждого вызова). Тир 3 — единственный, где скоуп-энфорс обходим
  (значение у модели), потому и минт.

**Скоуп виден модели в ОБОИХ движках — читаемым файлом, не CLI.** Под agent-vm
нашего `vault`-бинаря в госте НЕТ (F1/F9), поэтому `vault ls` там не выполнить.
Лончер кладёт в контекст сессии файл `VAULT-ACCESS.md` (что доступно, по каким
адресам, границы скоупа) — монтируется/впрыскивается и в bwrap, и в VM (F5/mount).
Это и есть прозрачность Р0 в VM. `vault ls` — удобство для bwrap/оператора, не
единственный канал.

### 4.5. Коннекторы (плагины Vault)

Скоуп «только эти документы/папки» требует ПОНИМАНИЯ сервиса (какие URL, где
в запросе fileId, как резолвится папка) — это не generic-код. Форма — как
модули оркестратора: реестр, «выключено = не существует».

Контракт коннектора (Python-протокол, набросок):

```python
class Connector(Protocol):
    name: str                                   # "gdocs", "jenkins", …
    def authorize(self, req: HttpReq, secret: Secret) -> HttpReq: ...
    #   подставить кред (заголовок/query/подпись — знает коннектор)
    def in_scope(self, req: HttpReq, scope: dict) -> Verdict: ...
    #   ALLOW | DENY(reason, remedy) | ASK(descr) — ASK поднимает хук «спроса».
    #   DENY.remedy ОБЯЗАТЕЛЕН и ПРЕДПИСЫВАЮЩИЙ (см. Р0): что не так + что
    #   делать вместо + что доступно (напр. «док вне скоупа; в скоупе папка
    #   Team/X (3 дока: …); нужен именно этот — попроси оператора»). Ведёт
    #   модель по единственному верному пути, гасит поиск обходов.
    #   «что доступно» берётся из КЭША скоупа сессии (лончер сложил при launch,
    #   тот же, что в VAULT-ACCESS.md) — БЕЗ живого вызова сервиса на каждый
    #   DENY (иначе вложенный отказ, если и этот вызов упадёт).
    def oauth_flow(self) -> OAuthFlow | None: ...       # `vault connect <svc>`
    def resolve_scope(self, human: dict) -> dict: ...    # "Team/X" → folder id
    def mint(self, scope: dict) -> Secret | None: ...    # тир 3, если умеет
    def refresh(self, secret: Secret) -> Secret | None: ...  # свой сервисный OAuth
```

Первые коннекторы: `generic-bearer` (fallback: подстановка + URL-префиксы —
покрывает сегодняшний «curl со скоупом»), затем `gdocs` (OAuth + скоуп по
docs/folders + deny на share/permissions/export вне скоупа), `jenkins`
(API-token + скоуп по job-путям + минт), `mediawiki`, `discourse`.

Policy (эволюция secrets.toml):

```toml
[secrets.gdocs-projectX]
connector = "gdocs"
sessions  = ["projectX-*"]          # fnmatch по имени сессии/задачи
scope     = { docs = ["<id>"], folders = ["<id>"] }
confirm   = false                   # true → каждый вызов через подтверждение
```

Сервисные OAuth-токены (gdocs и т.п.) Vault ХРАНИТ и рефрешит сам (это его
секреты, в отличие от Anthropic-профилей — см. 4.7). `vault connect gdocs` —
интерактивный OAuth-флоу коннектора.

### 4.6. Смешанная модель доступа (решение оператора)

- **Базовое** — в policy заранее; работает везде, в т.ч. standalone.
- **Расширение спросом** — `in_scope → ASK`: Vault поднимает хук
  «access-request» (HTTP-callback клиенту / очередь). Оркестратор рендерит его
  кнопками в Telegram/Web (переиспользуется существующий permission-relay):
  `[✅ этот док, на сессию] [📁 папку навсегда] [❌]`. Грант — эфемерный
  (память Vault, до конца сессии) или запись в policy.
  **ASK ОБЯЗАН иметь таймаут** (Р0 «никогда не повисать»): оператор не ответил
  за `ask_timeout` (дефолт напр. 120с) → запрос модели НЕ виснет на TCP, а
  завершается предписывающим DENY (`remedy`: «оператор не ответил вовремя;
  сделай X / попроси позже»). Иначе `curl` модели повиснет и упадёт невнятной
  сетевой ошибкой вместо внятного отказа.
  **Реализовано (фаза 4):** три исхода — `❌` / `✅` разово / `🔒 навсегда`.
  «Навсегда» пишет УЗКИЙ грант в policy (`scope.url_prefixes` ровно на
  запрошенный ресурс, без query, без «всего сервиса»), текст запроса показывает
  оператору точную будущую запись и команду отзыва (`/wallet scope <имя> -<url>`).
  Третья кнопка появляется, только если коннектор дал узкий грант И включён
  `WALLET_POLICY_EDIT`; иначе её нет вовсе и в тексте сказано, почему. Кнопкой
  владеет ASK кошелька: обычные подтверждения тулов остались двоичными.
  Живой прокси после записи расширяет свой scope сам (прокси других сессий
  подхватят файл при следующем подъёме).
- **Standalone**: attended (`claude-box` в tty) → вопрос в терминал `[y/N]`
  (разовый грант; `«навсегда»` в tty — ещё TODO, `TtyVaultHost`/`BoxVaultHost`
  третий исход не предлагают); unattended (`-p`, CI) → deny+log. Модель при
  DENY/ASK получает внятную ошибку (прозрачность для модели).
- Аудит: `vault log` / `vault status`; под оркестратором события дублируются
  в бабл (как wallet-use сейчас).

### 4.7. Anthropic-профили — указатели, не секреты

`vault profile add work` регистрирует ИМЯ → каталог кред
(`~/.claude-profiles/work` с `.claude/`-структурой; путь КОРОТКИЙ — под VM
`$HOME`-swap упирается в лимит unix-сокета 108 байт, F4). Vault НЕ хранит и НЕ
рефрешит OAuth-токены (F7): рефреш делает сам claude (bwrap) или кред-прокси
agent-vm (VM). Применение профиля — задача лончера (5.2). Policy может
привязывать профиль к задаче («projectX-* → work»).

⚠️ **Риск (обязателен живой smoke в фазе 3): куда agent-vm пишет обновлённый
OAuth при рефреше в VM?** Если в state-каталог cwd (`<hash>/claude/`, F5), а НЕ
в каталог профиля — источник F4-пересева (`$HOME/.claude`) остаётся со старым
одноразовым refresh-токеном → следующий запуск профиля пересеет отработанный
токен → **разлогин учётки оператора**. Проверить прямым тестом: профиль в VM →
дождаться рефреша → перезапуск тем же профилем; работает ли. Фолбэки, если
рефреш НЕ синхронизируется в профиль: (а) после сессии копировать
`<hash>/claude/.credentials.json` обратно в каталог профиля; (б) `$HOME`-swap
целиться прямо в каталог профиля (тогда agent-vm пишет в него же — но проверить,
не ломает ли это state-layout). Не оставлять на «они рефрешат — значит ок».

⚠️ **Коллизия профиль↔state при одном cwd (F5).** State-каталог = `hash(cwd)`,
от профиля НЕ зависит. Две задачи в ОДНОМ каталоге под РАЗНЫМИ профилями делят
один state/транскрипты → утечка контекста между учётками. Лончер: при смене
профиля в том же cwd — предупредить/изолировать (свой state-путь на профиль,
если agent-vm позволит через `AGENT_VM_STATE_DIR`), иначе документировать как
ограничение.

## 5. Слой 2: Launcher (`claude-box`)

### 5.1. UX (утверждён оператором)

```
claude-box init                      # визард: движок по умолчанию, профиль, vault
claude-box profile add work          # ссылка на учётку
claude-box connect gdocs             # проксирует в vault connect
cd ~/project && claude-box           # claude В ПЕСОЧНИЦЕ + кошелёк (дефолт)
claude-box --vm                      # то же в microVM — ЕДИНСТВЕННОЕ отличие
claude-box --profile work --vm
claude-box --wallet projectX -- --model opus    # скоуп + сквозные аргументы claude
claude-box -p 'задача'               # unattended (deny+log вместо спроса)
```

Ощущение: «как `claude`, только безопасно». Вся разница движков спрятана.

### 5.2. Что делает лончер под капотом (по движкам)

Общее: спросить Vault грант для задачи → получить session-токен Vault и
описание скоупа → собрать конфиг клиента (`.mcp.json` не нужен в standalone;
`settings.local.json` — hooks не нужны в standalone, но env/permissions — да)
→ запустить через Engine (Слой 0, runners как есть) → отдать PTY.

| Аспект | bwrap | agent-vm |
|---|---|---|
| профиль | `CLAUDE_CONFIG_DIR=<profile>` в env | `$HOME=<profile-home>` процессу agent-vm + сохранить `XDG_STATE_HOME` (F4) |
| env для claude | env процесса | `env`-блок settings (F1) |
| адрес перехвата Vault | `127.0.0.1:<port>` (HTTP_PROXY, наш MITM) | **без форка:** реверс-эндпоинт/base_url на `<LAN-IP>:<port>` + `--allow-egress` (F2, НЕ наш MITM — F10). **с форком:** egress-proxy на `<LAN-IP>:<port>` (полный MITM) |
| shims | PATH-обёртки (как сейчас) | НЕТ (git/gh ведёт agent-vm) |
| Anthropic base_url | как задан | LAN-IP + свой токен, иначе honest-отказ (F3) |
| транскрипты | `~/.claude/projects/...` (или профиль) | `state/<sha256(cwd)[:12]>/claude/projects/` (F5) |

Реализация НЕ с нуля: это вынос из `orchestrator/core/sessions.py` (PTY-запуск,
`_DialogAnswerer` авто-диалогов, готовность «по тишине», рендер настроек) и
`orchestrator/runners/*` — уже работающего и покрытого тестами кода.

### 5.3. Программный интерфейс для Слоя 3

Лончер = python-библиотека + CLI поверх неё. Оркестратор использует библиотеку:
`launch(task) -> Handle {pty_fd, transcript_path, stop(), resume(), clear()}`.
Каналы/хуки (channel_server, hook_dispatch) — это НАДСТРОЙКА оркестратора: он
докидывает свои `--mcp-config/--settings` при запуске через лончер.

## 6. Слой 3: Оркестратор (что меняется, что нет)

НЕ меняется: адаптеры Telegram/Web, транспорт-протокол, бабл, permission-relay,
turn-supervisor, история, channel_server, хуки — весь UX.

Меняется:
- `SessionManager` худеет: запуск/провижн/готовность уезжают в лончер; остаётся
  реестр сессий, привязки, состояние на диске.
- `modules/wallet` уезжает в Vault; модуль в оркестраторе остаётся ТОНКИМ
  клиентом (команды `/vault`, рендер access-request'ов кнопками, бабл-аудит).
- `features()`/`COMMAND_FEATURE` (единый реестр доступного — уже реализован)
  наполняется из capabilities лончера/Vault.
- `/stats` под agent-vm чинится чтением транскрипта по F5 (лончер отдаёт путь).

Правила проекта, обязательные и для нового кода:
- **Прозрачность для модели и оператора** (главное правило): отклонено →
  модель получает ошибку, не молчание; недоступно → оператору честный отказ
  с объяснением, а не тихий no-op.
- **Выключено = не существует**: никаких артефактов выключенной фичи в меню,
  `/help`, кнопках, промте.
- Требования к песочнице объявляет модуль/компонент (`MODULE_REQUIRES_SANDBOX`
  — паттерн сохранить).

## 7. Безопасность (модель угроз, решения зафиксированы)

- Секрет не входит в машину (перехват подставляет между машиной и сервисом) →
  прямой доступ модели к сервису без перехвата пуст (401), egress не ограничиваем.
  Скоуп гарантирован тем, что авторизованный путь один — через Vault.
- Тир 3 — только минт под-токена; мастер-секрет — только по ЯВНОМУ полю policy
  `allow_master` (отдельное от `confirm`, см. §4.4).
- Vault-демон: bind 127.0.0.1, bearer per-client, файлы policy 0600, значения
  секретов вымарываются из любого вывода/логов (перенести `_redact` из
  кошелька).
- Токены в `settings.local.json` (VM): файл 0600; от МОДЕЛИ это не защита
  (в госте она root) — поэтому туда только СВОИ прокси-токены Vault
  (плейсхолдеры/session-токены), never мастер-секреты.
- Не редактировать структуру state-каталога agent-vm (F5) — только добавление
  файлов; при обновлении agent-vm — smoke-тест.

## 8. Порядок реализации (фазы = вехи ветки)

Каждая фаза — рабочее состояние, тесты зелёные, вливается фича-ветками в
`redesign/claude-box` (см. регламент ниже).

1. **Vault-extract.** Вынести кошелёк в пакет `vault/` с самостоятельным
   демоном (CLI `vault`, HTTP API, policy TOML). Оркестратор — клиент через
   тонкий модуль (адаптер, поведение 1:1 с сегодняшним). Критерий: все
   wallet-тесты проходят против нового демона; `vault` работает без
   оркестратора.
2. **Перехват + generic-bearer (без форка — обход под VM).**
   - bwrap: exec-шим (расширить на HTTP-тулы) + MITM-прокси с CA для прочих
     клиентов. Критерий: `curl https://тест-сервис` с реальным URL, шим/MITM
     подставляет кред, запрос доходит авторизованным (спайк это подтвердил).
   - agent-vm: НЕ наш MITM (F10) — конфиг-редирект (base_url) + реверс-эндпоинт
     `http://<LAN-IP>/svc/…`. Критерий VM-плеча: модель ходит на наш
     реверс-эндпоинт (НЕ на произвольный HTTPS-URL — его перехватит их прокси),
     Vault авторизует наверх; доезжает. Переиспользовать существующий egress-код
     `orchestrator/config.py`/`runners/agentvm.py` (`agent_vm_host_ip`,
     `egress_hosts`, `--allow-egress`) до его переезда в лончер (фаза 3) — НЕ
     ждать фазу 3 и не дублировать.
3. **Launcher-extract.** Пакет `box/` (библиотека + CLI `claude-box`): вынос
   PTY-запуска/провижна/готовности из sessions.py; профили (bwrap:
   CLAUDE_CONFIG_DIR; VM: $HOME-swap, F4); транскрипт-путь (F5). Оркестратор
   переводится на библиотеку. Критерий: `claude-box` запускает интерактивный
   claude в обоих движках без оркестратора; все тесты оркестратора зелёные.
4. **Спрос (ASK-flow).** Хук access-request в Vault → рендер кнопками в
   оркестраторе (permission-relay), tty-вопрос в standalone, deny+log в
   unattended. Эфемерные гранты на сессию.
5. **Коннектор gdocs** (первый настоящий): OAuth-флоу, resolve_scope,
   скоуп-энфорс, deny на share/export. Затем jenkins (+минт), mediawiki,
   discourse — по мере надобности оператора.
6. **Полировка**: `/stats` в VM через F5, `vault log/status`, документация
   (README обоих новых пакетов, migration notes).

**Ф. Форк agent-vm `--egress-proxy` (ОТДЕЛЬНЫЙ трек, `FORK-agent-vm-egress-proxy.md`).**
Даёт паритет: под agent-vm перехват становится полным, как под bwrap (egress
гостя → наш Vault-MITM). НЕ блокирует фазы 1–6 (до него VM живёт на обходе).
Начинается с РАЗВЕДКИ сборки (собирается ли agent-vm из исходников; где проброс
флага) — до неё сроки не называть. Критерий: без флага agent-vm 1:1 как апстрим
(тест на регрессию); с флагом — наш Vault-прокси видит расшифрованный запрос
гостя и подставляет секрет (повтор спайка, но наш прокси уже НЕ обойдён).

Фазы 1–3 почти не пишут нового поведения — это перенос проверенного кода за
чистые интерфейсы. Новое поведение — фазы 2 (перехват), 4, 5 и трек Ф (форк).

## 9. Регламент работ (требование оператора)

- Долгоживущая ветка **`redesign/claude-box`**; фичи — ветками ОТ неё и PR-ами
  В неё; в `main` — только целые фазы (или финал), после живого smoke.
- Тесты вперёд (регламент рефакторинга проекта): каждый срез — сначала
  офлайн-тесты (pytest + запуск файлом, как весь `tests/`), затем код.
  Живые smoke по чек-листам `docs/agent-vm-integration.md`.
- Ревью Sonnet-агентом ДО PR (как принято в проекте), CI (ruff + pytest +
  run_all.sh на 3.10/3.12) обязателен для PR в redesign-ветку.
- Прод (main) живёт своей жизнью; критические фиксы — в main, redesign
  регулярно ребейзится/мержит main, чтобы не разъехаться.
- Коммиты без Claude-трейлеров (правило репозитория).
