# claude-box: принятые решения и их обоснования

Дополнение к `ARCHITECTURE-claude-box.md` (само задание — там). Здесь — ПОЧЕМУ
решено так, и какие альтернативы отклонены. Формат: решение → обоснование →
отклонённое. Все решения утверждены оператором 2026-07-21/22; не пересматривать
без него.

## Р1. Слоистая форма (Vault / Launcher / оркестратор-надстройка)

**Решение:** разделить монолит на слои; изолированный запуск claude с профилем
и секретами — самостоятельная технология, оркестратор — её клиент.

**Обоснование:** две постановки оператора — «запускать отдельные экземпляры
claude с моим профилем, URL и кошельком как в agent-vm, так и в песочнице» и
«текущий оркестратор должен стать надстройкой над этой технологией». Прообраз
формы — сам agent-vm (он и есть Engine+Vault+Launcher для узкого случая
Anthropic+gh, без оркестратора); мы обобщаем кред-слой на произвольные секреты
и делаем лончер движко-агностиком. Кошелёк оператор явно хочет «вытащить
отдельно от оркестратора».

**Отклонено:** продолжать наращивать монолит (кошелёк и запуск навсегда
привязаны к Telegram-боту — противоречит постановке).

## Р2. Прозрачный перехват (exec-шим / MITM-прокси), РЯДОМ с прокси agent-vm

**Решение:** модель ходит на НАСТОЯЩИЙ URL сервиса; перехват и подстановка креда
— прозрачны, между машиной и сервисом. Точка перехвата по движку: bwrap —
exec-шим (команда исполняется на хосте, TLS не трогается) для именованных тулов
+ MITM-прокси для прочих клиентов; agent-vm — MITM-прокси (HTTP_PROXY + CA
Vault в trust-store гостя). Клиенты с конфигом эндпоинта (Anthropic base_url) —
конфиг-редирект без MITM. Под agent-vm наш перехват РЯДОМ с их кред-прокси
(Anthropic/gh ведут они), не вместо.

**Обоснование:** (1) **Ревью-1 ошибочно предложило «модель ходит на наш
localhost-эндпоинт» — оператор указал: модель знает реальный URL из обучения и
использует его, заставить выбрать наш эндпоинт нельзя.** Значит перехват обязан
быть прозрачным. (2) Прозрачно подставить кред в HTTPS-запрос к реальному URL
можно либо exec-шимом (перехват на команде, не на сети — но только именованные
тулы), либо TLS-MITM (терминировать TLS своим CA — универсально, любой клиент).
MITM — **основной механизм**, и он ДОКАЗАННО работает в этой среде: ровно так
делает кред-прокси самого agent-vm (MITM + свой CA в госте). cert-pinning ломает
лишь меньшинство (браузеры, мобильные SDK), не curl/API/SDK. (3) «рядом» — их
OAuth-refresh Anthropic труден в повторе, а целевые сервисы оператора закрывает
наш перехват.

**Отклонено:** (а) «модель выбирает наш localhost-эндпоинт» — нельзя, модель
использует реальный URL (ошибка ревью-1, снята оператором); (б) «вместо» их
прокси — тянет OAuth-refresh+gh. MITM НЕ отложен (ранее ошибочно счёл хрупким) —
он основной; конфиг-редирект — там, где клиент это поддерживает (надёжнее,
без MITM).

## Р3. Форк agent-vm отклонён; свой лончер на msb отклонён

**Решение:** agent-vm используем как есть (плюс, опционально, PR-ы в апстрим).

**Обоснование:** исследованы шесть путей — использовать найденные точки
расширения; свой кред-прокси; свой OCI-образ; апстрим-PR; форк; свой лончер
поверх msb. Форк — это Rust + сборка нативных бинарей (патченный msb, libkrunfw)
и вечная догонялка за активным апстримом ради 2–3 флагов. Свой лончер на msb —
ещё дороже: теряются их кред-прокси, gh-прокси, egress-политика и OAuth-refresh,
всё пришлось бы переписать. При этом эксперименты показали, что нужное
достижимо БЕЗ форка (Р4). Апстрим-PR (`-e KEY=VAL` — механизм env в
SandboxConfig уже есть, флага нет; `--claude-profile`) — желателен, но не
блокер: до него живём на точках расширения.

**Отклонено:** форк (дорого, догонялка), свой msb-лончер (дороже форка),
кастомный OCI-образ как ОСНОВНОЙ путь (не решает креды; остаётся опцией для
тяжёлых кастомизаций гостя).

## Р4. Точки расширения agent-vm — используем, с оговоркой

**Решение:** опираемся на два недокументированных, но проверенных живьём
механизма: (а) state-каталог VM — обычная хостовая папка
`~/.local/state/agent-vm/<sha256(cwd)[:12]>/` (гостевой `/root/.claude` —
симлинк внутрь неё: транскрипты читаемы с хоста, файлы хоста видны в госте);
(б) креды сеются из `$HOME/.claude` процесса agent-vm при каждом запуске →
выбор учётки = запуск с подменённым `$HOME`.

**Обоснование:** (а) чинит транскрипты/stats в VM и проброс профиля без каких-
либо доработок agent-vm; (б) единственный работающий способ выбора учётки
(`CLAUDE_CONFIG_DIR` agent-vm игнорирует — замерено; подкладывание в
`.secrets` бесполезно — пересев подтверждён по mtime).

**Оговорка (в архитектуре зафиксирована):** поведение недокументировано — в
state-каталог только ДОБАВЛЯТЬ файлы (одна попытка записи сломала оверлей msb),
smoke-тест при каждом обновлении agent-vm.

## Р5. Anthropic-профиль — указатель на учётку, НЕ секрет в Vault

**Решение:** Vault хранит имя→каталог кред; сам OAuth-токен Claude не хранит и
не рефрешит. Применяет профиль лончер (bwrap: `CLAUDE_CONFIG_DIR`; VM:
`$HOME`-swap).

**Обоснование:** сервис-секреты статичны, а клод-учётка — живой OAuth с
одноразовой ротацией refresh-токена. «Профиль как секрет» втянул бы в Vault
refresh+ротацию+запись назад — то, что сегодня бесплатно и корректно делают сам
claude (bwrap) и прокси agent-vm (VM). Учётки оператора — подписочные claude.ai
(OAuth), не API-ключи — проверено по структуре кред-файла.

**Отклонено:** хранить живой токен в Vault (реимплементация OAuth, риск
рассинхрона одноразовых refresh-токенов = разлогин настоящего клода).

## Р6. Выдача секрета: прозрачный перехват по умолчанию, реальное значение — только минт

**Решение (уточнено после ревью):** прежние «тир 1 подстановка на проводе» и
«тир 2 реверс-прокси» СЛИЛИСЬ: значение в машину не входит, перехват
прозрачен (exec-шим/MITM — см. Р2). Escape hatch — «тир 3»: реальное значение только как
минт короткоживущего/скоупнутого под-токена; мастер-секрет — лишь по ОТДЕЛЬНОМУ
флагу policy `allow_master` (НЕ перегружать `confirm` — тот про подтверждение
каждого вызова).

**Обоснование:** запрос оператора «дать модели логин/пароль или токен
контролируемый». Прозрачный перехват даёт контроль без выдачи значения вовсе; тир 3 нужен
для веб-форм/не-HTTP и честно ограничен: где сервис умеет под-токены (Jenkins,
GitLab) — минт, где нет — осознанное решение оператора, не дефолт. Тир 3 —
единственный, где скоуп-энфорс обходим (значение у модели), поэтому и минт.

## Р7. Egress не ограничиваем

**Решение:** прямой сетевой доступ модели к сервисам (например `api.google.com`)
не блокируем.

**Обоснование:** формулировка оператора: «токен не подставлять для прямого
обхода — и пусть что хочет без него делает». Мерить надо не «куда дотянется по
сети», а «где есть креды»: при прозрачном перехвате секрета в машине нет,
прямой запрос к сервису БЕЗ перехвата получает 401 — путь открыт, но пуст. Гарантия скоупа —
единственность авторизованного пути через перехват Vault, а не сетевые заборы.
Связь с механизмом: это верно ИМЕННО потому, что дефолт — прозрачный перехват (Р2/Р6), где
значение не входит в машину; для тира 3 (минт) обход возможен — потому и
только ограниченный под-токен.

**Отклонено:** ранняя версия дизайна требовала блокировать egress к доменам
сервисов — ошибка в модели угроз, снята.

## Р8. Vault модульный: ядро + сервис-коннекторы

**Решение:** знания о конкретных сервисах — в плагинах-коннекторах (контракт в
архитектуре, §4.4); ядро — хранение/policy/транспорты/аудит. Fallback —
`generic-bearer` (подстановка + URL-префиксы, покрывает сегодняшний «curl со
скоупом»).

**Обоснование:** задача «gdocs со скоупом по документам/папкам» недоказуема
generic-кодом: bearer-подстановка отдаёт токен учётки целиком (Google не минтит
под-токены на папку), значит скоуп режет наш прокси, а для этого надо понимать
Drive API (fileId из URL, parents-резолв, опасные эндпоинты share/export). Это
знание сервиса = плагин. Модульность повторяет паттерн оркестратора (реестр,
«выключено = не существует») — оператор про это сказал прямо: «не зря делал
модульный оркестратор».

## Р9. Доступ: базовое в конфиге + расширение «спросом»; спрос — фича надстройки

**Решение:** типовой скоуп прошивается в policy заранее; выход за него —
поведение НАСТРАИВАЕТСЯ (см. ниже), по умолчанию модель запрашивает в моменте,
оператор решает кнопками (гранты: на сессию/навсегда). Vault лишь экспонирует
хук access-request; рендер кнопок — оркестратор (существующий permission-relay).

**Поведение при выходе за скоуп — настройка коннектора/политики, а не жёсткое
правило (уточнение оператора 2026-07-22).** Поле `out_of_scope` в policy-записи
(и дефолт коннектора, и глобальный дефолт):

- `ask` — поднять access-request (доставка: tty если attended-терминал, кнопки
  если оркестратор);
- `deny` — сразу отказать, БЕЗ повисания, с внятной ошибкой модели (что вне
  скоупа и что политика — deny). Работает и в интерактивной сессии: оператор
  может хотеть авто-отказ, не вопрос.

Разрешение = policy-запись → дефолт коннектора → глобальный дефолт. Attended-
ность влияет ТОЛЬКО на доставку `ask`; решение ask/deny — из настройки. Если
`ask`, но канала спросить нет (unattended standalone) → деградация в `deny+log`
(спросить некого), с той же внятной ошибкой модели.

```toml
[secrets.gdocs-projectX]
connector    = "gdocs"
scope        = { folders = ["<id>"] }
out_of_scope = "deny"     # ask | deny  (нет поля → дефолт коннектора → глобальный)
```

**Обоснование:** выбор оператора «смешанно» из трёх моделей (только конфиг /
только спрос / смешанно), плюс требование управлять поведением по умолчанию
даже в интерактиве: «хочу иметь возможность автоматом отказать без повисания с
понятной ошибкой; а где-то — чтобы запрос дошёл до меня». Ложится на слои без
нарушения автономности: Vault про чат не знает; переиспользуется готовый relay.

## Р10. UX лончера: один флаг движка

**Решение:** `claude-box [--vm] [--profile X] [--wallet Y] [-- аргументы claude]`
— переезд bwrap↔VM меняет ровно один флаг, вся разница механизмов спрятана.

**Обоснование:** требование оператора «как пользователь запускает claude в
песочнице и в VM с кошельком — и там и там, до оркестратора». Вся асимметрия
движков (env vs settings-блок, loopback vs LAN-IP, shims vs их прокси,
CLAUDE_CONFIG_DIR vs $HOME-swap) — внутренняя таблица лончера (архитектура,
§5.2), пользователю не видна. «Ощущение как `claude`, только безопасно».

## Р11. Регламент: отдельная долгоживущая ветка

**Решение:** редизайн живёт в `redesign/claude-box`; фичи — ветками от неё,
PR в неё, тесты вперёд; в main — только целые фазы после живого smoke.

**Обоснование:** требование оператора («это почти полностью переписывает
проект»); прод на main продолжает жить и получать фиксы независимо.

## Р0. Прозрачность — сквозное требование всех слоёв (не пункт, а принцип)

**Решение:** каждое действие и каждый отказ прозрачны И для оператора, И для
модели — обе стороны понимают ЧТО произошло и ПОЧЕМУ. Это надрешение: оно
подчиняет себе Р6/Р9 и всё будущее поведение, а не отдельная фича.

**Для модели — отказ ПРЕДПИСЫВАЮЩИЙ, а не просто причина (тезис оператора
2026-07-22):** «если модели не просто отказать, а сказать, что она сделала не
так и что надо делать, она перестанет тратить время на обходные пути и пойдёт
по единственно верному». Поэтому каждый DENY несёт ТРИ части:
1. **что не так** — «док вне выданного скоупа»;
2. **что делать вместо** — «доступ к нему запрашивается у оператора»/«используй
   завёрнутый инструмент X»/«это в скоупе — просто зови как обычно»;
3. **что доступно сейчас** — «в скоупе: папка Team/X (3 дока: …)».

Это средство (`Verdict.DENY.remedy`, обязательное — архитектура §4.4),
и оно знание сервиса → ответственность коннектора. **Двойная выгода:**
модель идёт по верному пути И перестаёт искать обход (base64/другой
эндпоинт/выудить креды/retry-циклы) — а это ровно тот флейлинг, от которого
защищается `WALLET_GUARD`. Предписывающий отказ ДЕШЕВЛЕ и БЕЗОПАСНЕЕ голого.

Прочее для модели:
- никогда не молчание/повисание/пустой результат — всегда ошибка инструмента;
- свой скоуп виден (`vault ls`): что доступно и в каких границах;
- доступ идёт через прозрачный перехват — значение секрета модель не видит
  (перехват подставляет его между машиной и сервисом).

**Для оператора:**
- аудит фиксирует не только факт, но и ОСНОВАНИЕ решения: выдано политикой /
  одобрено в рантайме / отказано (по какой настройке) — `vault log`, живьём в
  бабл под оркестратором;
- недоступная/выключенная возможность — честный отказ с объяснением, а не тихий
  no-op (правило проекта «выключено = не существует»).

**Обоснование:** прямое требование оператора (2026-07-22): «прозрачность везде
важна, и для оператора, и для модели — обе стороны должны понимать, что
происходит и почему». Совпадает с зафиксированным ранее главным правилом
проекта (поведение прозрачно для модели и оператора; отклонено → ошибка, не
молчание). Вынесено в Р0, потому что это критерий приёмки КАЖДОЙ фазы, а не
отдельный пункт.
