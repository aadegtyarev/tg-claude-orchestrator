# Транспорт-коннектор: путь к Matrix (набросок)

Короткий дизайн-набросок — не план работ. Цель на будущее: добавить Matrix
как второй канал связи, не переписывая ядро. Сейчас сознательно ничего не
абстрагируем «впрок» (это крупный редизайн — обсуждаем отдельно), но
декомпозиция уже движется в нужную сторону.

## Что уже транспорт-независимо

После рефакторинга 2026-07 от aiogram зависят только `bot.py` и `bubble.py`.
Остальное Telegram не знает:

- `sessions.py` / `runner.py` / `transcript.py` — процессы и изоляция;
- `turn.py` — жизненный цикл хода (колбэки `send(thread_id, text)` и
  `typing(thread_id)` уже абстрактные);
- `toolline.py`, `cbdata.py`, `logsignals.py`, `mdrender.py` — чистый рендер
  и разбор (mdrender целится в HTML-подмножество Telegram; у Matrix тоже
  HTML-разметка — почти совместимо);
- `channel_server.py`, `reply_server.py` — протокол с Claude;
- `texts.py` — каталог сообщений.

## Соответствие понятий

| Оркестратор | Telegram | Matrix |
|-------------|----------|--------|
| Сессия | форум-топик группы | комната (или thread в комнате) |
| Статус-бабл | редактируемое сообщение | событие + `m.replace`-правки |
| Кнопки (permission, стоп) | inline-кнопки | реакции/reply-команды (нативных кнопок нет) |
| Реакции-ack (👀/👍) | reactions | reactions |
| «печатает…» | chat action | typing notification |
| Белый список | user id | MXID |

Самое неудобное в Matrix — отсутствие inline-кнопок: permission relay и
«⏹ Стоп» придётся делать реакциями (✅/❌ на сообщение-запрос) или
текстовыми ответами. Это решаемо, но UX-развилки стоит прожить заранее.

## Направление работ (когда дойдём)

1. Выделить из `bot.py` интерфейс `Transport`: `send`, `edit`, `delete`,
   `react`, `typing`, `ask_buttons` (у Matrix — эмуляция реакциями), события
   `on_message/on_file/on_button`. `TelegramTransport` = текущий код.
2. Командный слой (`/new`, `/stats`, роутинг сообщений) — общий, поверх
   Transport; у транспортов остаются только их родные особенности.
3. `MatrixTransport` на matrix-nio; конфиг выбирает один или оба транспорта
   (одна сессия видна из одного транспорта — маппинг топик/комната ↔ сессия).

Принцип наблюдаемости сохраняется: статус-бабл и события инструментов должны
работать в каждом транспорте, а не только в Telegram.
