# Видение: изолированный запуск claude с профилем и секретами — отдельно от оркестратора

Статус: **план, не реализация** (2026-07-22). Фиксирует направление, поставленное
оператором: кред-слой (профиль, URL, кошелёк) — самостоятельная технология,
работающая в agent-vm И в песочнице БЕЗ оркестратора; текущий оркестратор
становится надстройкой над ней.

## Зачем (реальные задачи оператора)

Дать модели доступ к сервисам: **MediaWiki, Jenkins, Discourse** и т.п. — без
выдачи ей самих токенов. Это тот же кошелёк, но в масштабе «много сервисов» и в
обоих движках изоляции. Плюс: разные учётки Claude (профиль/URL) на разные
задачи.

## Ключевое наблюдение, упрощающее всё

**MediaWiki / Jenkins / Discourse — это HTTP-сервисы.** Значит их секреты
закрываются ОДНИМ механизмом — подстановкой токена на проводе (host-side
proxy: гость шлёт плейсхолдер, хост подставляет реальное значение), и этот
механизм работает **одинаково в bwrap и в agent-vm**. Проверено живьём, что
сетевой путь до хостового прокси из VM есть (LAN-IP + `--allow-egress` + свой
токен, PR #48).

Отсюда — два транспорта секрета, а не один:
- **wire-proxy (HTTP)** — универсален для ОБОИХ движков; покрывает Anthropic,
  MediaWiki, Jenkins, Discourse — всё, что ходит по HTTP. Это и есть отложенная
  «TLS-прокси с подстановкой» из secrets-wallet.md. Изоляцию НЕ ослабляет
  (секрет не входит в гостя, канала «гость → исполнитель на хосте» нет).
- **shims (exec)** — обёртки в PATH для git/ssh/scp. Только bwrap (в VM PATH/env
  наши не доходят). Не-HTTP секреты (ssh-ключи) — только здесь.

## Слои (целевая архитектура)

```
Слой 3  ОРКЕСТРАТОР        сессии, Telegram/Web, бабл, resume — UX поверх Слоя 2
                            (клиент Launcher, НЕ знает про креды/секреты)
        ─────────────────────────────────────────────────────────────
Слой 2  LAUNCHER           «запусти агента X в движке Y с профилем/политикой Z
                            в каталоге C» — самостоятельный CLI. По сути это
                            «обобщённый agent-vm»: движок-агностик.
        ─────────────────────────────────────────────────────────────
Слой 1  VAULT              хостовый демон: держит секреты и профили, отдаёт их
        (секреты+профиль)   через два транспорта (wire-proxy / shims) + policy
                            «какой сессии/задаче какой секрет». Автономен.
        ─────────────────────────────────────────────────────────────
Слой 0  ENGINE (изоляция)  bwrap | agent-vm | off. Уже есть: runners/.
```

Две идеи оператора ложатся на слои:
- «запускать экземпляры claude с профилем/URL/кошельком в обоих движках» =
  Слой 2 (Launcher) поверх Слоя 1 (Vault) и Слоя 0 (Engine).
- «оркестратор — надстройка над этой технологией» = Слой 3 поверх Слоя 2.

Это ровно та форма, что у самого agent-vm (он = Слои 0–2 для Anthropic+gh, без
оркестратора). Мы обобщаем Слой 1 на произвольные секреты и делаем Слой 2
движок-агностиком.

## Что уже готово к этой форме

- **Слой 0 готов**: `runners/` — движок-агностик шов (bwrap|agent-vm|off).
- **Слой 1 частично есть**: `modules/wallet/` — демон+policy+shims. НО вшит в
  оркестратор (`core.session_hooks`, `manager.env_hooks/path_hooks`, пишет
  `~/.wallet.json` в дом сессии). Извлечение = сделать его самостоятельным
  демоном с клиентской библиотекой; и Launcher, и оркестратор — его клиенты.
  Сейчас у кошелька есть shims, но НЕТ wire-proxy — его надо добавить.
- **Сетевой путь для wire-proxy в VM проверен** (PR #48).

## Две РАЗНЫЕ задачи, которые сейчас спутаны (расцепить — главная ценность плана)

- **A. Какая учётка Claude** (профиль/креды/URL). bwrap: `CLAUDE_CONFIG_DIR`
  работает уже. agent-vm: `$HOME`-своп (проверено — креды сеются из
  `$HOME/.claude`) ИЛИ наш прокси вместо ихнего (дорого — их OAuth-refresh).
- **B. Произвольные сервис-секреты** (MediaWiki/Jenkins/Discourse). Оба движка:
  наш wire-proxy (HTTP) + shims (bwrap, не-HTTP).

Это разные проблемы, случайно связавшиеся через «профиль». A — про Anthropic-
эндпоинт, B — про чужие сервисы. Разводим и решаем по отдельности.

## Главная развилка: рядом с прокси agent-vm или вместо него

- **Рядом (рекомендую).** Под agent-vm их прокси продолжает вести Anthropic+gh
  (включая OAuth-refresh — он реально полезен и труден в воспроизведении). НАШ
  wire-proxy добавляется РЯДОМ и ведёт только произвольные сервис-секреты (B).
  Гость достаёт наш прокси уже проверенным путём. Мало работы, не реимплементим
  OAuth. Под bwrap конкурента нет — наш слой ведёт всё.
- **Вместо.** Наш прокси заменяет их для всего. Чище концептуально, но тянет
  реимплементацию OAuth-refresh и gh — большая работа и хрупкость. Не сейчас.

Практический вывод: **для задач оператора (MediaWiki/Jenkins/Discourse) хватает
«рядом»** — и тогда спайк OAuth-проксирования НЕ нужен вовсе. Выбор учётки (A)
под agent-vm решаем `$HOME`-свопом отдельно, не через замену прокси.

## Anthropic-профиль: указатель, а не секрет в Vault (решено 2026-07-22)

Считать профиль «ещё одним секретом» — концептуально проще, практически
сложнее: сервис-секреты СТАТИЧЕСКИЕ (положил значение — подставляй), а Anthropic
OAuth ЖИВОЙ (протухает, refresh-токен ротируется одноразово). Vault-как-хранилище
статичных значений тогда обязан вести OAuth-refresh+ротацию+запись назад.
Сейчас рефреш ведёт сам claude (bwrap) или прокси agent-vm (VM) — «руки прочь».

**Решение:** Vault держит УКАЗАТЕЛЬ («задаче T положена учётка = такой
creds-каталог/$HOME»), а жизненный цикл токена остаётся у claude/agent-vm.
Единообразие политики без реимплементации OAuth. Живой токен в Vault не кладём.

## Прокидывание секрета В машину: три тира (решено 2026-07-22)

Различать, что нужно модели, а не «дать секрет» бинарно:

1. **Значение НЕ нужно** (исходящий HTTP к сервису) → wire-proxy подставляет на
   проводе, модель шлёт плейсхолдер. Секрет в машину не входит. Лучший случай.
2. **Нужен доступ, не секрет** → авторизующий реверс-прокси: в машину выставлен
   `localhost/<сервис>`, прокси сам авторизуется наверху реальным секретом.
   Модель получает рабочий эндпоинт и НОЛЬ кредов. «Контролируемый доступ».
3. **Нужно реальное значение** (веб-форма в браузере, не-HTTP инструмент) →
   отдаём, но можем выдать МИНТ короткоживущего/скоупнутого под-токена вместо
   мастер-секрета (Jenkins API-token, GitLab PAT, OAuth exchange). «Токен
   контролируемый»: реальный, но ограниченный и отзываемый.

Граница: тир 3 требует, чтобы сервис умел выдавать под-токены; где не умеет —
либо полный секрет (доверие модели), либо тир 2. Это мощнее нынешнего кошелька
(бинарное inject/shared): добавляются «доступ без секрета» и «ограниченное
значение».

Оба вывода работают в ОБОИХ движках (тиры 1–2 — по сети через wire-proxy,
проверено PR #48; тир 3 — выдача значения, через env-блок настроек в VM или
env процесса в bwrap).

## Модульность Vault: сервис-коннекторы (докид оператора, 2026-07-22)

Оркестратор модульный не зря — Vault должен быть таким же. Типовая задача,
которая это доказывает: **работа с Google Docs со скоупом «только эти
документы/папки»**.

**Почему тиров 1–3 недостаточно.** Bearer-подстановка (тир 1) отдаёт запросу
токен учётки ЦЕЛИКОМ — Google-токен открывает всё, до чего дотянется учётка.
«Ограничить конкретными документами» токеном не выразить (Google не выдаёт
под-токены на папку — тир 3 недоступен). Значит скоуп должен резать НАШ слой —
авторизующий реверс-прокси (тир 2), который смотрит В КАЖДЫЙ запрос.

**А чтобы смотреть в запрос, надо понимать сервис.** «Разрешены документы X,Y и
папка Z» — это знание о том, как устроен Drive API: из каких URL достаётся
fileId, как проверить принадлежность папке (parents-цепочка), какие эндпоинты
опасны (share/permissions/export). Это НЕ generic-код. Это плагин.

**Форма: Vault = ядро + сервис-коннекторы** (модульность как в оркестраторе —
`modules/`, реестр, «выключено = не существует»):

```
vault-core        хранение секретов, policy, wire-proxy/shims, аудит, минт
коннектор gdocs   OAuth-обвязка Google + скоуп-энфорсер: fileId из запроса,
                  parents-резолв для папок, deny на share/export вне скоупа
коннектор mediawiki   токен + скоуп по неймспейсам/страницам (если надо)
коннектор jenkins     API-токен + скоуп по job-путям; минт под-токенов (тир 3)
коннектор discourse   api-key + скоуп по категориям
generic-bearer    fallback без знаний: подстановка + URL-префиксы (нынешний
                  «curl со скоупом commands» из secrets-wallet.md)
```

Контракт коннектора (набросок): `авторизуй(запрос) → запрос'` (подставить
кред), `в_скоупе?(запрос, policy) → да/нет/спросить`, опционально
`минт(policy) → под-токен` и `рефреш()`. Policy в secrets.toml получает
секцию скоупа, специфичную коннектору:

```toml
[secrets.gdocs-projectX]
connector = "gdocs"
sessions  = ["projectX-*"]
scope     = { docs = ["<id1>", "<id2>"], folders = ["<folderZ>"] }
```

**Почему прямой доступ к сервису НЕ дыра (снято оператором, 2026-07-22).**
Первая версия этого раздела требовала закрывать прямой egress к доменам
сервиса — это была ошибка в модели угроз. Мерить надо не «куда гость дотянется
по сети», а «где у него есть креды»: в тирах 1–2 секрет в машину не входит
вообще, прямой запрос модели к `api.google.com` идёт БЕЗ токена и получает
401. Прямой путь открыт, но пуст — обходить нечего. Гарантия скоупа: секрет
существует только на нашем прокси, и он не подставляет кред в запросы, идущие
мимо policy. Egress-блокировки не нужны; неавторизованная сетевая активность
модели — вне нашей задачи (public-интернет ей и так доступен).

Оговорка остаётся ТОЛЬКО для тира 3 (модели выдано реальное значение): с ним
она может пойти к сервису напрямую, мимо скоуп-энфорса. Поэтому тир 3 — всегда
минт ОГРАНИЧЕННОГО под-токена, а не мастер-секрета; где сервис минт не умеет —
осознанное решение оператора (policy `confirm`/доверие), не дефолт.

Тогда сегодняшний кошелёк раскладывается без потерь: host-passthrough/inject →
vault-core (тир 1 + shims), «curl с URL-скоупом» → generic-bearer, TLS-прокси
из «отложено» → wire-proxy ядра, а новое (gdocs-скоуп, минт) — коннекторы.

## UX Слоя 2: как человек запускает claude изолированно (ДО оркестратора, 2026-07-22)

Это фундамент. Лончер полезен САМ ПО СЕБЕ — человек запускает claude безопасно,
без всякого оркестратора. Оркестратор — опциональная надстройка. Рабочее имя
CLI ниже — `claude-box` (имя за оператором).

**Главная задача лончера: bwrap и agent-vm выглядят ОДИНАКОВО.** Всю разницу
механизмов (LAN-IP, `--allow-egress`, env-блок vs env процесса, shims vs
wire-proxy, профиль через $HOME vs CLAUDE_CONFIG_DIR) лончер прячет. Пользователь
меняет ОДИН флаг движка — profile/wallet/поведение идентичны.

### Настройка (разово)

```
claude-box init                 # визард: движок по умолчанию, профиль(и), кошелёк
claude-box profile add work     # учётка (OAuth-логин/URL) — «work»
claude-box connect gdocs        # OAuth к сервису (см. S0), кладётся в кошелёк
```

### Запуск — то, ради чего всё (одинаково для обоих движков)

```
cd ~/project
claude-box                          # = claude, но В ПЕСОЧНИЦЕ (дефолт) + кошелёк
claude-box --vm                     # то же, но в microVM (agent-vm)
claude-box --profile work --vm      # другая учётка, в VM
claude-box --wallet projectX -- --model opus   # скоуп кошелька + аргументы claude
```

`--vm` против дефолта (bwrap) — ЕДИНСТВЕННОЕ, что пользователь меняет, чтобы
переехать между движками. Ощущение — как `claude`, только безопасно.

### Внутри сессии

- **Человек за терминалом** (attended): обычный TUI claude; кошелёк прозрачен
  (модель зовёт инструменты как всегда); вне скоупа → запрос ПРЯМО В ТЕРМИНАЛ:
  `🔐 модель просит gdocs «Budget 2026» — разрешить? [y/N/навсегда]`.
- **Без человека** (`claude-box -p '…'`, CI/cron): вне скоупа → **deny+log**
  (спросить некого). Это и есть уточнение S5: attended → tty-запрос,
  unattended → deny+log.

### Два фронтенда одного лончера

Лончер экспонирует PTY с claude. К нему подключается:
- **человек** — напрямую в терминале (этот раздел, standalone);
- **оркестратор** — программно (Слой 3), добавляя сессии, Telegram/Web, бабл,
  resume. Много лончеров, driven, с чат-UX.

Один примитив, два потребителя. Поэтому «оркестратор — надстройка»: он не
запускает claude сам, он ДРАЙВИТ лончер. Сегодняшняя логика sessions.py
(PTY+раннер+провижн) — это зачаток лончера, вросший в оркестратор; вынести =
Слой 2.

## UX кошелька: модель доступа «смешанная» — базовое в конфиге, расширение спросом (2026-07-22)

Оператор выбрал: типовой доступ прошит заранее (config), выход за границы модель
просит в момент, оператор подтверждает кнопками, грант — на сессию (или навсегда).

**Структурное следствие (ложится на слои):**
- **Базовое (config)** живёт в Vault — работает и в standalone, и под оркестратором.
- **«Спрос»** — фича ОРКЕСТРАТОРА (Слой 3): нужен канал к оператору
  (Telegram/web permission-relay, УЖЕ есть). Vault лишь ЭКСПОНИРУЕТ хук «нужен
  доступ вне скоупа»; кто сверху — решает, как спросить. Standalone без
  оркестратора: «спрос» деградирует в deny+log (или `--interactive` → tty).
  Vault про чат не знает — автономность сохранена.

### Сквозные сценарии (UX как истории)

**S0. Оператор подключает сервис (разово).** `/vault connect gdocs` → бот шлёт
OAuth-URL Google → оператор авторизует в браузере → «✅ gdocs подключён как
user@x». Токен в Vault, рефреш автоматом (OAuth-обвязка коннектора).

**S1. Оператор объявляет базовый доступ (config).**
`/vault grant projectX-* gdocs --folder "Team/ProjectX"` → коннектор резолвит имя
папки → id (discovery, чтобы не копировать id руками) → «выдано: сессии
projectX-* видят папку Team/ProjectX (3 дока)». Или правкой secrets.toml.

**S2. Модель работает В скоупе (прозрачно, без оператора).** Сессия projectX-app.
Модель зовёт gdocs-инструмент/эндпоинт как обычно; Vault подставляет auth на
проводе, коннектор режет скоуп. Модель видит ТОЛЬКО папку Z. В её контексте
(промт + `vault ls`, который она может позвать): «доступ на чтение/запись к папке
Team/ProjectX; другие файлы не видны». Бабл: «🔓 gdocs: прочитан 'Q3 Planning'».
Ноль участия оператора.

**S3. Модель хочет ВНЕ скоупа (спрос).** Пытается открыть док вне папки Z.
Коннектор возвращает модели «denied: вне скоупа» (чтобы она знала) И поднимает
запрос доступа. Оператору в чат:
```
🔐 projectX-app просит Google Doc
   «Budget 2026» (вне выданного скоупа)
   [✅ Этот док, на сессию] [📁 Добавить его папку навсегда] [❌ Отказать]
```
Тап → грант на сессию (эфемерный, модель повторяет и проходит) / запись в config
навсегда / отказ (модель адаптируется). Кнопки — тот же relay, что permission.

**S4. Оператор аудитирует.** `/vault log projectX-app` → «14:03 read gdocs
'Q3 Planning'; 14:05 DENIED 'Budget 2026'; 14:06 grant(session) оператором».
Живьём — в бабл сессии (как wallet-use сейчас). `/vault status` — что кому
доступно прямо сейчас.

**S5. Standalone (без оркестратора).**
`vault-launch --engine bwrap --grant gdocs:projectX -- claude …` — запуск claude
с базовым скоупом. Вне скоупа: чата нет → **deny+log** по умолчанию, либо
`--interactive` → подтверждение в tty. Standalone = базовое, если не запрошен
интерактив. Честная деградация «спроса».

### Что переиспользуется, что новое

Переиспользуется: permission-relay (кнопки S3), бабл-аудит (S4),
`/wallet`→`/vault` команды (S0–S1), провижн секрета в сессию. Новое: OAuth-
обвязка коннектора (S0), discovery id по имени (S1), скоуп-энфорс в запросе
(S2–S3), хук «запрос доступа» из Vault наружу (S3), standalone-launcher (S5).

### Где поставлен дефолт (скажи, если не так — НЕ блокирующий вопрос)

- Гранулярность кнопок S3 («док / папка / сессия / навсегда») — предложено.
- Standalone вне скоупа = deny+log (не tty по умолчанию) — безопаснее.
- Модель ВИДИТ свой скоуп (`vault ls`) — по правилу прозрачности для модели.

   (Launcher, оркестратор) ходят по локальному сокету/HTTP с токеном?
   (склоняюсь к да — это и есть «кошелёк без оркестратора».)
2. **Формат политики:** оставить TOML `secrets.toml` или обобщить на «задача →
   сервисы» со скоуп-секцией коннектора (см. пример в разделе модульности).
3. **Совместимость:** оркестратор сейчас поднимает кошелёк in-process. Миграция
   на внешний Vault — поэтапно (адаптер к обоим), не big-bang.

## Что НЕ делаем в этой итерации

Не строим. Не форкаем agent-vm (см. agent-vm-extension-options.md: форк дорог,
«рядом»-прокси его обходит). Не спайкуем OAuth (рекомендация «рядом» его
снимает). Следующий шаг — согласовать слои и развилку «рядом/вместо», затем
резать на атомарные срезы по регламенту рефакторинга.
