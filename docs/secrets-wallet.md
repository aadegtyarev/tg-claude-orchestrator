# Кошелёк секретов: токены для CLI-тулз без доступа модели к значениям

**Статус: реализовано** (`orchestrator/modules/wallet/`, включение `MODULES=wallet`).
Работает ТОЛЬКО вместе с `SANDBOX=bwrap` — см. [«Зачем песочница»](#зачем-песочница-без-bwrap-кошелёк-бесполезен).

Задача: инструменты, которые Claude запускает в сессии (`gh`, `git push`,
`kubectl`, `curl` к API, `ssh`…), должны работать со своими токенами/паролями/
ключами, но **модель не должна видеть сами значения** — ни прочитать файл, ни
`echo $TOKEN`, ни выудить промпт-инъекцией и отправить в чат.

Как это устроено: секреты лежат в одном файле на хосте (вне песочницы); команда
с секретом исполняется демоном **на хосте**, а модель получает только результат
(значения вымараны из вывода). CLI `wallet` доступен в PATH сессии; демон —
localhost-сервис в составе оркестратора, аутентификация per-session токеном в
`~/.wallet.json` приватного дома сессии. Тесты: `tests/wallet_test.py`,
`tests/wallet_policy_test.py`, `tests/wallet_inject_test.py`.

## Быстрый старт

1. `MODULES=wallet` в `.env` (и `SANDBOX=bwrap` — дефолт).
2. Файл секретов `~/.config/claude-orchestrator/secrets.toml`, права строго
   `0600` (путь меняется `WALLET_SECRETS_FILE`):

```toml
[secrets.host]
description = "хостовые креды gh/git/ssh/scp"
sessions = ["*"]                        # fnmatch по имени сессии; нет поля = никому
commands = ["gh", "git", "ssh", "scp"]  # голое имя = любой вызов инструмента
confirm = false                         # без кнопок; guard — щит, вызов виден строкой
```

3. В сессии: `wallet ls` (что доступно) · `wallet run host -- gh pr list`.

Правку файла демон подхватывает на ходу (читает лениво по mtime) — **рестарт не
нужен**. Policy можно смотреть и править прямо из Telegram — `/wallet` (см.
[ниже](#правка-policy-из-бота-wallet)).

## Зачем песочница (без bwrap кошелёк бесполезен)

Единственная ценность кошелька — конфиденциальность секретов от модели, а она
держится на песочнице. `SANDBOX=bwrap` прячет от сессии сырые креды хоста:
`~/.ssh`, `~/.config/gh`, `~/.aws`, keyring, чужой `$HOME`, env-переменные с
токенами. Кошелёк отдаёт обратно *контролируемый* доступ — исполняет команду на
хосте, значение в песочницу не попадает.

Без песочницы (`SANDBOX=off`/`agent-vm`) модель читает все хостовые креды
напрямую (`cat ~/.config/gh/hosts.yml`, `echo $TOKEN`) — прятать нечего, барьера
нет. Поэтому провижн `~/.wallet.json` завязан на bwrap-дом сессии; под другими
раннерами CLI просто не найдёт конфиг, а модуль громко предупреждает на старте.

## Два режима секрета

| | host-passthrough | inject |
|---|---|---|
| запись в `secrets.toml` | **без** `value`/`env` | `value` + `env` |
| где креды | на хосте (keyring, `~/.config/gh`, `~/.aws`, git auth, `~/.ssh`) | значение в самом `secrets.toml` |
| как работает | команда идёт на хост с ЕГО окружением | значение в env-переменную процесса + плейсхолдеры в команду |
| для чего | `gh`, `git push`, `kubectl`, `aws`, `docker`, `ssh`/`scp` по ключам хоста — уже залогинены | «сырой» токен, которого нет на хосте: `gh`/`aws`/`kubectl` (читают env) или `curl`/ssh-ключ (плейсхолдеры) |
| редакция вывода | не нужна (значения нет) | да: точные вхождения `value` → `•••` |

**host-passthrough — дефолт и самый сильный** (значение вообще не существует в
конфиге, живёт в keyring/credential-helper хоста; `echo`/`cat` показывать
нечего). Пиши запись **без** `value`/`env` — команда исполнится на хосте с его
кредами. Годится для всего, что уже авторизовано на хосте.

## Токен в аргумент или файл: плейсхолдеры (curl, ssh-ключи)

Команда под секретом исполняется **без шелла** (`exec`), поэтому `$ENV` в
аргументе НЕ развернётся — inject-env годится лишь для инструментов, читающих
переменную сами (`gh`→`GH_TOKEN`, `aws`, `kubectl`, `docker`). Когда токен
должен попасть **в строку аргумента** (заголовок curl) или **в файл** (ssh-ключ,
клиентский сертификат, JSON-креды) — есть плейсхолдеры, которые демон
подставляет **у себя**, перед запуском; значение в контекст модели и в песочницу
не попадает, а из вывода редактируется:

| плейсхолдер | во что превращается | пример |
|---|---|---|
| `{{secret}}` | значение прямо в аргумент | `curl -H "Authorization: Bearer {{secret}}" https://api/...` |
| `{{secret_file}}` | путь к временному `0600`-файлу со значением (сносится после команды) | `ssh -i {{secret_file}} user@host` · `curl --cert {{secret_file}}` |

Оба — только для inject-секретов (у host-passthrough значения нет; попытка → отказ).
Файл пишется в хостовый `/tmp`, который песочница не видит (`--tmpfs /tmp`).
(Прежний совет `sh -c 'curl … $VAR'` убран: он требовал давать `sh` и полагался
на разворот `$VAR` шеллом.)

⚠️ Для curl **сужай `commands` по URL** (`commands = ["curl https://api.trusted/*"]`),
иначе модель сможет вставить `{{secret}}` в запрос на произвольный адрес и
вынести токен наружу. Подстановка прячет значение от модели, но не решает, куда
уходит запрос — это делает allow-лист `commands`.

### SSH — три случая

1. **Ключ уже на хосте (`~/.ssh`)** — host-passthrough, плейсхолдеры не нужны:
   `wallet run host -- ssh user@host` (`ssh`/`scp` в дефолтном списке команд).
   Модель ключ не видит, ssh читает его сам.
2. **Приватный ключ как секрет** (в кошельке, не на хосте) — inject `value` =
   тело ключа, и `ssh -i {{secret_file}} user@host`: демон материализует ключ во
   временный `0600`-файл, подставит путь, удалит после.
3. **Пароль** — inject `env = "SSHPASS"` + `sshpass -e ssh user@host` (sshpass
   читает `$SSHPASS` из env; ставится отдельно). `sshpass -p {{secret}}` не
   советуем — пароль виден в `ps` на хосте.

## Как модель выбирает нужный секрет из нескольких

Каталог секретов приходит в системный промпт при старте сессии (и печатается
`wallet ls`), и по каждому показывает **имя + описание + разрешённые команды
(включая URL)**. Модель матчит цель с описанием/URL и берёт нужное **имя** —
`wallet run <имя> -- …`, один запуск = один секрет. Пример с токенами к разным
сайтам:

```toml
[secrets.stripe]
description = "Stripe API (prod)"
value = "sk_live_…"
env = "STRIPE_KEY"
commands = ["curl https://api.stripe.com/*"]

[secrets.openai]
description = "OpenAI API"
value = "sk-…"
env = "OPENAI_KEY"
commands = ["curl https://api.openai.com/*"]
```

Модель видит `• stripe (Stripe API (prod)) — curl https://api.stripe.com/*` и
`• openai (OpenAI API) — curl https://api.openai.com/*`, и для Stripe берёт
`wallet run stripe -- curl -H "Authorization: Bearer {{secret}}" https://api.stripe.com/…`.
Ошиблась именем → `commands` не того секрета не пропустят чужой URL → отказ,
fail-safe. От тебя нужно лишь дать каждому **внятное `description`** и **сузить
`commands` под его сайт**.

## Развилка: что у модели в руках, что в кошельке

Кред, который у модели **легитимно на руках** — дал юзер в чате, или она сама
сгенерила под НОВЫЙ ресурс — она использует напрямую обычными командами (`sshpass`
с этим паролем, свой `ssh-keygen` + `ssh -i`). Кошелёк тут не при делах: он
подменяет только то, что модель видеть **не должна** (хостовые gh/git/ssh, токен
из списка). Одно ограничение: не заимствовать/подделывать ХОСТОВУЮ identity в
обход кошелька (`gh auth login`, токен в git-remote, правка хостовых
`~/.ssh`/`~/.config/gh`) — для этого `wallet run` или эскалация оператору.

Локальный git без кредов (`status`/`diff`/`add`/`commit`) идёт напрямую — через
кошелёк только сетевые операции.

## Секрет для СЕРВИСА, который пишет модель — это не кейс кошелька (host/inject)

Отдельный частый случай: модель написала сервис (напр. заметки с AI-редактором),
и **сервису** в рантайме нужен ключ (OpenAI и т.п.). Спрятать этот ключ от модели
через host/inject **невозможно в принципе**: сервис написала она, код под её
контролем — она всегда может заставить его раскрыть значение
(`print(os.environ["KEY"])`, запись в файл проекта, отправка в сеть). Кошелёк в
режимах host/inject прячет значение лишь от команд, **чей код модель не
контролирует** (`gh`, `git`, `curl`).

Поэтому для сервиса разводим два момента:

| | кому нужен ключ | как |
|---|---|---|
| **Dev** (модель тестирует сервис) | самой модели | мок, дешёвый throwaway-ключ с лимитом, или проверка формы запроса через `wallet run` |
| **Runtime/prod** (сервис сам зовёт API) | процессу сервиса | deploy-секрет: оператор даёт при запуске (systemd `EnvironmentFile` 0600, docker/k8s secret), сервис читает из env |

Модель пишет сервис так, чтобы он читал ключ из env (`OPENAI_API_KEY`), **никогда
не хардкодит** и не коммитит; строку кладёт в `.env.example`/README. Боевой ключ
живёт только в деплое, вне сессии модели. Dev-ключ модель всё равно видит — его
достаточно сделать дешёвым/лимитированным. Держать такой dev-ключ (или логин/
пароль) удобно в кошельке как **shared-секрет** (см. ниже), чтобы не палить в
чате/репо.

## Режим `shared` — секрет, который модель ВИДИТ (dev-ключ, логин/пароль)

Иногда модель ДОЛЖНА получить значение: dev-ключ для сервиса, который она пишет;
логин/пароль, которые она вводит в браузере/форме. Прятать нечего — но хранить
такое удобно в кошельке: **не палить в чате, не коммитить в репо**, держать в
одном месте (0600, хост), ротировать, гейтить по сессиям. Это НЕ про
конфиденциальность от модели (значение она видит), а про хранение и доставку.

```toml
[secrets.openai_dev]
description = "OpenAI dev key (лимитированный)"
shared = true               # значение ВЫДаётся сессии
value = "sk-…"
env = "OPENAI_API_KEY"       # опц.: имя переменной для env-выдачи/авто-инъекта
inject_at_start = true       # опц.: положить env=value в окружение сессии на старте
sessions = ["*"]
confirm = false              # опц.: спрашивать кнопкой перед выдачей
```

Выдача (в сессии):

- `wallet get openai_dev` — печатает **значение** (вставить, ввести в форму);
- `wallet env openai_dev` — печатает `export OPENAI_API_KEY='…'`; модель делает
  `eval "$(wallet env openai_dev)"` или кладёт в `.env` сервиса;
- `inject_at_start = true` — переменная `env` кладётся в окружение процесса
  `claude` при старте; её наследуют Bash-тул модели и сервисы, что она запускает
  → сервис просто читает `OPENAI_API_KEY`, без действий модели. (Новый
  shared-секрет подхватится после resume/рестарта сессии.)

Границы: значение выдаёт ТОЛЬКО `shared` — host/inject значения не отдаются
никогда (эндпоинт `/get` их отклоняет). `commands`/`guard`/`deny` к shared не
применяются (команды нет). Гейтится `sessions`/`confirm`, выдача видна строкой в
статус-бабле. **Только для дешёвых/лимитированных/dev-кредов** — модель значение
видит, guard его не страхует; боевые ключи — в деплой, не в shared.

## Настройка policy — справочник

**Принцип.** Кошелёк не должен душить модель (иначе проще прогать самому),
поэтому по умолчанию удобно: голое имя инструмента = любой его вызов. Но есть
пол, который держится всегда, — само условие смысла кошелька: модель должна
**использовать** креды, не **читая** их. Три слоя, в порядке применения:

1. **allow — `commands`** (где кошелёк доступен). Голое имя (`gh`, `git`, `ssh`) —
   любой вызов инструмента; строка с пробелом/глобом (`gh api repos/*`,
   `git push*`) — fnmatch по всей команде. Пусто у host-passthrough → дефолт
   `["gh","git","ssh","scp"]`; пусто у inject → ничего (сырой токен не открываем
   без явного списка).
2. **guard — встроенный всегда-запрет** (поверх allow). Рубит опасное независимо
   от `commands`, с **прозрачным объяснением модели** (что не так и как правильно
   — доходит до её терминала): команды, печатающие сам токен (`gh auth token`,
   `gh auth status --show-token`), и git-RCE через флаги/транспорт (`git -c …`,
   `ext::…`, `--receive-pack`/`--upload-pack`/`--exec`). Отключается глобально
   `WALLET_GUARD=0` или точечно на секрет `allow_unsafe = true`.
3. **deny — точечный запрет секрета** (поверх allow, deny побеждает). Голый токен
   (`--force`, `--hard`) блокирует этот флаг/аргумент где угодно; строка с
   пробелом/глобом — fnmatch по всей команде. Для «разрешаю инструмент, но не эти
   опасные опции».

**Поля записи `[secrets.<имя>]`:**

| поле | смысл | дефолт |
|---|---|---|
| `value` + `env` | inject: значение и env-переменная для него | нет = host-passthrough |
| `description` | человекочитаемое; помогает модели выбрать нужный секрет | пусто |
| `sessions` | fnmatch имён сессий, кому доступен (`*` = все) | нет = никому |
| `commands` | allow-лист (голые имена/шаблоны) | host: `gh git ssh scp`; inject: ничего |
| `deny` | точечный запрет поверх allow | нет |
| `allow_unsafe` | снять встроенный guard на этом секрете | `false` |
| `confirm` | кнопки ✅/❌ перед запуском (и перед выдачей shared) | `true` |
| `shared` | значение ВЫДаётся сессии (`wallet get`/`env`); см. «Режим shared» | `false` |
| `inject_at_start` | для shared: положить `env`=value в env сессии на старте | `false` |

**Что по умолчанию** (`MODULES=wallet` + запись без лишних полей):
host-passthrough с `gh/git/ssh/scp`, guard включён, `confirm=true`. Хватает для
«дай сессии работать с git/gh хоста, не пуская токен в модель».

**Примеры:**

```toml
# Хостовые креды для всех сессий (keyring/gh/git/ssh) — удобно и безопасно:
[secrets.host]
sessions = ["*"]
commands = ["gh", "git", "ssh", "scp"]   # любой вызов; guard рубит опасное

# Разрешить инструмент, но запретить разрушительные флаги:
[secrets.careful-git]
sessions = ["ci-*"]
commands = ["git"]
deny = ["--force", "--force-with-lease", "git push --delete*"]

# Тонко: только чтение через gh api:
[secrets.readonly]
sessions = ["report"]
commands = ["gh api user", "gh api repos/*"]

# Доверенный случай, где нужен как раз git -c (guard мешает):
[secrets.special]
sessions = ["build"]
commands = ["git"]
allow_unsafe = true    # ОСОЗНАННО снимаем guard на этом секрете
confirm = true
```

## Правка policy из бота (`/wallet`)

Policy можно смотреть и править из Telegram, не открывая файл на хосте:

- `/wallet` — список секретов и policy в код-блоке (**значения токенов не
  показываются**) + подсказка по правке прямо в ответе;
- `/wallet confirm <имя> on|off` — спрашивать ли кнопкой перед запуском;
- `/wallet session <имя> +<шаблон>|-<шаблон>` — кому доступен (`*`, `dev-*`);
- `/wallet cmd <имя> +<шаблон>|-<шаблон>` — какие команды (`gh`, `git`, `gh api *`);
- `/wallet deny <имя> +<шаблон>|-<шаблон>` — точечный запрет поверх commands;
- `/wallet new <имя>` — создать host-passthrough секрет (пустой, deny-by-default);
- `/wallet rm <имя>` — удалить секрет.

Правки пишутся в `secrets.toml` атомарно (tmp+`os.replace`, права 0600
сохраняются, комментарии не теряются — `tomlkit`) и подхватываются демоном по
mtime, **без рестарта**.

Границы: **значения токенов** (inject `value`/`env`) и `allow_unsafe` через бота
не трогаются — только правкой host-файла (иначе токен утёк бы в историю чата).
Команда только для `ALLOWED_USER_IDS`; модель вызвать её не может (приходит от
пользователя, не из Claude). Глобальный тумблер **`WALLET_POLICY_EDIT`** (по
умолчанию включён; `0` — только просмотр, правки лишь host-файлом).

## Слои защиты — единая связка «песочница + кошелёк»

Конфиденциальность держится не на одном барьере, а на слоях, которые **не
работают порознь**:

1. **Песочница (`SANDBOX=bwrap`) — фундамент.** Прячет от сессии сырые креды
   хоста. Без неё модель читает всё напрямую — остальное теряет смысл.
2. **Кошелёк — контролируемый доступ обратно.** Даёт применять креды, не читая
   их. Барьеры: `sessions`/`commands`/`deny`/`confirm` + встроенный guard.
3. **Судья auto-режима + `permissions.deny` — блок попыток в обход.** В
   `PERMISSION_MODE=auto` действия сессии оценивает отдельный классификатор
   Claude Code (в 2.1.2x по умолчанию Sonnet 5); его промпт не кастомизируется, а
   правила — да, оркестратор пишет их в `settings.local.json` сессии:
   - `autoMode.hard_deny` («не добывай секреты; для git/gh — кошелёк»),
     `autoMode.environment` (где лежат секреты), `classifyAllShell` (проверять
     ВСЕ bash-команды; из `AUTOMODE_CLASSIFY_ALL_SHELL`, дефолт on);
   - `permissions.deny` — жёсткое чтение кред-файлов (`~/.config/gh`, `~/.netrc`,
     `~/.ssh`, `~/.aws`, keyring-CLI). Работает во ВСЕХ режимах, включая `bypass`.

**Ключевое:** слой 3 добавляется ТОЛЬКО когда активна связка — `MODULES=wallet`
**и** `SANDBOX=bwrap`. Без песочницы правила были бы театром, без кошелька нечего
защищать. Всё это включается только целиком.

**Защита — механика, не уговоры в промпте.** Под bwrap хостовые креды физически
недостижимы для модели, а единственная точка, где значение секрета течёт к ней —
вывод `wallet run` — редактируется в демоне (значение → `•••`). Подсказка в
промпте лишь **учит пользоваться** (как выбрать секрет, как передать значение,
развилка владения, «не смог — скажи оператору») — она не барьер.

## Модель угроз и известные ограничения

Bwrap прячет `~/.ssh`, `~/.aws` и прочий дом, но всё, что мы *дали бы* сессии для
работы CLI-тулзы (env, `~/.netrc`, файл конфига), модель по определению может
прочитать. Значит секрет не должен быть в адресном пространстве песочницы вообще
— только результат его применения. Отсюда — остаточные дыры, которые
закрываются **доверием (policy), а не аргументами**:

- **Команда исполняется в cwd проекта, куда модель пишет.** `wallet run deploy --
  git push` запускает git на хосте в папке проекта; модель может подложить туда
  `./.git/config` (`core.sshCommand`/hooks) и получить исполнение на хосте с
  секретом в env. Узкий шаблон команды это НЕ закрывает; для inject демон
  выставляет `GIT_CONFIG_NOSYSTEM`/`GIT_CONFIG_GLOBAL=/dev/null` (частично).
  **Настоящий барьер — `sessions` (доверенные) + `confirm` (человек видит
  команду) + host-passthrough (значения нет).**
- **Редакция вывода — literal-only, best-effort.** Заменяются точные вхождения
  значений; любая трансформация в команде (base64, reverse, запись в файл)
  обходит. Это гигиена против случайного эха, не барьер.
- **Шаблоны команд — fnmatch по `" ".join(cmd)`.** `*` пересекает границы argv и
  пробелы. Пиши шаблоны консервативно (префикс + явные аргументы), не полагайся
  на `*` для ограничения хвоста.
- **Поддержан только `SANDBOX=bwrap`** (см. «Зачем песочница»).

Значения секретов (inject/shared) вымарываются не только из вывода `wallet run`,
но и из **исходящего в чат текста** — reply модели, служебных уведомлений и строк
статус-бабла (`OrchestratorCore._scrub` через редакторы модулей). Safety-net:
случайный `echo` shared-значения или запись ключа в `.env` не улетят в Telegram/
историю. Не барьер (модель значение держит), гигиена.

## Что отложено

- **MCP-тул `wallet_run`** (то же, что CLI, но нативно в permission-механизме
  Claude Code) — не реализован; сейчас интерфейс — CLI `wallet` (`ls`/`run`).
- **TLS-прокси с плейсхолдерами** (подмена токена в исходящем запросе, чтобы
  значение вообще не попадало на хост-команду) сильнее по конфиденциальности, но
  отложен: TLS-MITM хрупок (cert-pinning ломает его в ноль), покрывает только
  HTTP-секреты (не SSH-ключи/сертификаты), и дёшев только на microVM. Вводим
  вместе с переходом на `agent-vm` ([agent-vm-integration.md](agent-vm-integration.md)),
  где слой достаётся почти бесплатно (там так уже делается с OAuth-токенами
  Claude) — тогда он встанет на место inject для HTTP-токенов. Не «лучше вместо»,
  а «следующий слой, когда будет microVM».
