# Кошелёк секретов: токены для CLI-тулз без доступа модели к значениям

**Статус: реализовано** (`orchestrator/modules/wallet/`, включение — `SANDBOX_BWRAP_WALLET`,
по умолчанию ВКЛЮЧЁН при `SANDBOX=bwrap`).
Работает ТОЛЬКО вместе с `SANDBOX=bwrap` — см. [«Зачем песочница»](#зачем-песочница-без-bwrap-кошелёк-бесполезен).

Задача: инструменты, которые Claude запускает в сессии (`gh`, `git push`,
`kubectl`, `curl` к API, `ssh`…), должны работать со своими токенами/паролями/
ключами, но **модель не должна видеть сами значения** — ни прочитать файл, ни
`echo $TOKEN`, ни выудить промпт-инъекцией и отправить в чат.

**Главный принцип — прозрачность.** Модель зовёт инструмент **как обычно**
(`gh pr create`, `git push`, `curl -H "Authorization: Bearer $TOKEN" …`,
`ssh -i $KEY_FILE …`). Никакого специального синтаксиса знать не нужно: обёртка в
`PATH` песочницы молча заворачивает вызов в кошелёк, а секрет подставляется на
хосте. Раньше модель «не догадывалась» дёргать `wallet run` — теперь дёргать
нечего, всё работает само.

Как это устроено под капотом:

- секреты лежат в одном файле на хосте (вне песочницы, `0600`);
- в `PATH` сессии первым стоит каталог **per-session обёрток** `.wallet-bin`:
  для каждого разрешённого инструмента там лежит крошечный скрипт-обёртка;
- обёртка вызывает `wallet exec <tool> …` → демон **на хосте** подбирает секрет по
  команде, исполняет её с секретом в окружении/аргументах и возвращает вывод
  (значения секретов из него вымараны);
- inject-секреты видны модели как привычные env-переменные `$NAME`; в песочнице
  там **маркер**, реальное значение подставляет демон на хосте — модель его не
  видит.

CLI `wallet` доступен в `PATH` (ручной путь и справка); демон — localhost-сервис
оркестратора, аутентификация per-session токеном в `~/.wallet.json` приватного
дома сессии. Тесты: `tests/wallet_test.py`, `tests/wallet_policy_test.py`,
`tests/wallet_inject_test.py`, `tests/wallet_shared_test.py`,
`tests/wallet_shims_test.py`.

## Быстрый старт

1. Ничего не делать: при `SANDBOX=bwrap` (дефолт) кошелёк **включён по
   умолчанию**. Выключается `SANDBOX_BWRAP_WALLET=false`; эта же настройка сама
   объявляет модуль, так что `MODULES=wallet` писать не нужно (legacy-форма ещё
   работает, но при конфликте побеждает `SANDBOX_BWRAP_WALLET`).
   **`SANDBOX=bwrap` здесь обязателен, а не желателен:** провода кошелька (шимы
   первыми в PATH + env-маркеры) — это окружение процесса `claude`, поэтому под
   `SANDBOX=agent-vm` (claude в microVM, наш env туда не попадает) модуль не
   включается даже явно, с WARNING в логе. Что доступно в VM-режиме вместо него
   — см. docs/agent-vm-integration.md.
2. Файл секретов `~/.config/claude-orchestrator/secrets.toml` (путь меняется
   `WALLET_SECRETS_FILE`). **Если его нет — создаётся автоматически при первом
   запуске** с дефолтом «из коробки» (права `0600`):

```toml
[secrets.host]
description = "хостовые креды gh/git/ssh/scp"
sessions = ["*"]                        # fnmatch по имени сессии; нет поля = никому
commands = ["gh", "git", "ssh", "scp"]  # голое имя = любой вызов инструмента
confirm = false                         # без кнопок; guard — щит, вызов виден строкой
```

   То есть из коробки gh/git/ssh/scp во всех сессиях идут через
   кошелёк на хост с его кредами. Сузь `sessions`/`commands` или добавь секреты —
   правкой файла (права держи `0600`) или из бота `/wallet`.
3. В сессии модель просто зовёт `gh pr list`, `git push`, `ssh user@host` —
   обёртка направит через кошелёк. `wallet ls` показывает, что доступно.

Правку файла демон подхватывает на ходу (читает лениво по mtime) — **рестарт не
нужен**. Набор обёрток пересобирается при провижне сессии (старт/resume/рестарт);
после правки policy новой сессии обёртки уже соответствуют. Policy можно смотреть
и править прямо из Telegram — `/wallet` (см. [ниже](#правка-policy-из-бота-wallet)).

## Зачем песочница (без bwrap кошелёк бесполезен)

Единственная ценность кошелька — конфиденциальность секретов от модели, а она
держится на песочнице. `SANDBOX=bwrap` прячет от сессии сырые креды хоста:
`~/.ssh`, `~/.config/gh`, `~/.aws`, keyring, чужой `$HOME`, env-переменные с
токенами. Кошелёк отдаёт обратно *контролируемый* доступ — исполняет команду на
хосте, значение в песочницу не попадает.

Без песочницы (`SANDBOX=off`/`agent-vm`) модель читает все хостовые креды
напрямую (`cat ~/.config/gh/hosts.yml`, `echo $TOKEN`) — прятать нечего, барьера
нет. Поэтому провижн `~/.wallet.json` и обёртки завязаны на bwrap-дом сессии; под
другими раннерами CLI просто не найдёт конфиг, а модуль громко предупреждает на
старте.

## Три режима секрета

| | host-passthrough | inject | shared |
|---|---|---|---|
| запись в `secrets.toml` | **без** `value`/`env` | `value` + `env` | `shared=true` + `value` (+ опц. `env`) |
| где креды | на хосте (keyring, `~/.config/gh`, `~/.aws`, git auth, `~/.ssh`) | значение в самом `secrets.toml` | значение в `secrets.toml` |
| видит ли модель значение | нет | **нет** (маркер в песочнице) | **да** (значение выдаётся) |
| как модель пользуется | зовёт инструмент как обычно; креды берутся с хоста | зовёт инструмент как обычно; `$NAME`/`$NAME_FILE` в команде | `wallet get`/`wallet env`, или готовый `$NAME` в env |
| для чего | `gh`, `git push`, `kubectl`, `aws`, `ssh`/`scp` по ключам хоста — уже залогинены | «сырой» токен, которого нет на хосте: `gh`/`aws` (читают env), `curl`-заголовок, ssh-ключ | dev-ключ для сервиса модели, логин/пароль для ввода в браузере |

**host-passthrough — дефолт и самый сильный** (значение вообще не существует в
конфиге, живёт в keyring/credential-helper хоста; `echo`/`cat` показывать
нечего). Пиши запись **без** `value`/`env` — команда исполнится на хосте с его
кредами. Годится для всего, что уже авторизовано на хосте.

**inject** — для токена, которого на хосте нет: он лежит в кошельке. Модель видит
его как обычную переменную `$NAME`, но реальное значение живёт только на хосте
(см. следующий раздел).

**shared** — для значения, которое модель ДОЛЖНА получить; конфиденциальности от
модели тут нет, только хранение вне чата/репо (см.
[«Режим shared»](#режим-shared--секрет-который-модель-видит-dev-ключ-логинпароль)).

## Как inject-токен попадает в команду: env-переменные `$NAME`

Для каждого inject/shared-секрета с полем `env` модель получает **привычную
переменную окружения** `$NAME`. Использовать её нужно ровно так, как модель уже
умеет:

- **инструмент читает переменную сам** — `gh`→`$GH_TOKEN`, `aws`→`$AWS_…`,
  `kubectl`, `docker`: просто зови инструмент, env уже проставлен;
- **значение в строке аргумента** (заголовок curl) —
  `curl -H "Authorization: Bearer $OPENAI_KEY" https://api.openai.com/…`;
- **значение нужно файлом** (ssh-ключ, клиентский сертификат, JSON-креды) — есть
  парная переменная `$NAME_FILE`: `ssh -i $DEPLOY_KEY_FILE user@host`,
  `curl --cert $CERT_FILE`. Демон материализует значение во временный `0600`-файл
  на хосте (песочнице невидим, `--tmpfs /tmp`), подставит путь, удалит после.

Механика (модели знать не обязательно): в песочнице `$NAME` содержит не значение,
а **маркер** `<<wallet:имя>>` (а `$NAME_FILE` — `<<wallet:имя:file>>`). Когда
обёртка отправляет команду демону, тот разворачивает маркеры в реальное значение
уже **на хосте**, перед запуском. Значение в адресное пространство песочницы и в
контекст модели не попадает, а из вывода редактируется.

> ⚠️ Для `curl` **сужай `commands` по URL** (`commands = ["curl https://api.trusted/*"]`),
> иначе модель сможет подставить токен в запрос на произвольный адрес и вынести
> его наружу. Разворот маркера прячет значение от модели, но не решает, куда уходит
> запрос — это делает allow-лист `commands`.

### SSH — три случая

1. **Ключ уже на хосте (`~/.ssh`)** — host-passthrough, ничего подставлять не
   надо: `ssh user@host` (`ssh`/`scp` в дефолтном списке команд). Модель ключ не
   видит, ssh читает его сам.
2. **Приватный ключ как секрет** (в кошельке, не на хосте) — inject `value` =
   тело ключа, `env = "DEPLOY_KEY"`, и `ssh -i $DEPLOY_KEY_FILE user@host`: демон
   материализует ключ во временный `0600`-файл, подставит путь, удалит после.
3. **Пароль** — inject `env = "SSHPASS"` + `sshpass -e ssh user@host` (sshpass
   читает `$SSHPASS` из env; ставится отдельно). `sshpass -p $SSHPASS` не советуем
   — пароль виден в `ps` на хосте.

## git: сеть через кошелёк, локальное — напрямую

`git`-обёртка особая. Сетевые подкоманды (`push`, `fetch`, `pull`, `clone`,
`ls-remote`, `send-pack`, `fetch-pack`) уходят на хост через кошелёк (креды
хоста), а всё локальное (`status`, `add`, `commit`, `diff`, `log`, `branch`…)
бежит **настоящим git прямо в песочнице** — быстро, без хостового раунд-трипа и
без секрета. Модель во всех случаях зовёт просто `git …`.

## Как модель выбирает нужный секрет из нескольких

При `wallet exec` (через обёртку) демон подбирает секрет **по команде** сам:
берёт первый секрет, чьи `sessions` подходят сессии и чьи `commands` разрешают
эту команду. Поэтому для разных сайтов достаточно **сузить `commands` по URL** —
модель просто зовёт `curl …`, а нужный токен подставится по адресу:

```toml
[secrets.stripe]
description = "Stripe API (prod)"
value = "sk_live_…"
env = "STRIPE_KEY"
commands = ["curl https://api.stripe.com/*"]

[secrets.openai]
description = "OpenAI API"
value = "sk-…"
env = "OPENAI_KEY"
commands = ["curl https://api.openai.com/*"]
```

`curl -H "Authorization: Bearer $STRIPE_KEY" https://api.stripe.com/…` подберёт
секрет `stripe` (совпал URL), `…$OPENAI_KEY … https://api.openai.com/…` — `openai`.
Ошиблась переменной/адресом — `commands` не того секрета не пропустят чужой URL →
отказ, fail-safe. От тебя нужно лишь дать каждому **внятное `description`** (оно
попадает в каталог промпта и `wallet ls`) и **сузить `commands` под его сайт**.

Явно назвать секрет можно и вручную — `wallet run <имя> -- <cmd>` (один запуск =
один секрет); это ручной/отладочный путь, в норме обёртки его прячут.

## Развилка: что у модели в руках, что в кошельке

Кред, который у модели **легитимно на руках** — дал юзер в чате, или она сама
сгенерила под НОВЫЙ ресурс — она использует напрямую обычными командами (`sshpass`
с этим паролем, свой `ssh-keygen` + `ssh -i`). Кошелёк тут не при делах: он
подменяет только то, что модель видеть **не должна** (хостовые gh/git/ssh, токен
из списка). Одно ограничение: не заимствовать/подделывать ХОСТОВУЮ identity в
обход кошелька (`gh auth login`, токен в git-remote, правка хостовых
`~/.ssh`/`~/.config/gh`) — это делает кошелёк, а не модель.

## Секрет для СЕРВИСА, который пишет модель — это не кейс host/inject

Отдельный частый случай: модель написала сервис (напр. заметки с AI-редактором),
и **сервису** в рантайме нужен ключ (OpenAI и т.п.). Спрятать этот ключ от модели
через host/inject **невозможно в принципе**: сервис написала она, код под её
контролем — она всегда может заставить его раскрыть значение
(`print(os.environ["KEY"])`, запись в файл проекта, отправка в сеть). Кошелёк в
режимах host/inject прячет значение лишь от команд, **чей код модель не
контролирует** (`gh`, `git`, `curl`).

Поэтому для сервиса разводим два момента:

| | кому нужен ключ | как |
|---|---|---|
| **Dev** (модель тестирует сервис) | самой модели | мок, дешёвый throwaway-ключ с лимитом; удобно держать как `shared`-секрет |
| **Runtime/prod** (сервис сам зовёт API) | процессу сервиса | deploy-секрет: оператор даёт при запуске (systemd `EnvironmentFile` 0600, docker/k8s secret), сервис читает из env |

Модель пишет сервис так, чтобы он читал ключ из env (`OPENAI_API_KEY`), **никогда
не хардкодит** и не коммитит; строку кладёт в `.env.example`/README. Боевой ключ
живёт только в деплое, вне сессии модели. Dev-ключ модель всё равно видит — его
достаточно сделать дешёвым/лимитированным и хранить как `shared`.

## Режим `shared` — секрет, который модель ВИДИТ (dev-ключ, логин/пароль)

Иногда модель ДОЛЖНА получить значение: dev-ключ для сервиса, который она пишет;
логин/пароль, которые она вводит в браузере/форме. Прятать нечего — но хранить
такое удобно в кошельке: **не палить в чате, не коммитить в репо**, держать в
одном месте (0600, хост), ротировать, гейтить по сессиям. Это НЕ про
конфиденциальность от модели (значение она видит), а про хранение и доставку.

```toml
[secrets.openai_dev]
description = "OpenAI dev key (лимитированный)"
shared = true               # значение ВЫДаётся сессии
value = "sk-…"
env = "OPENAI_API_KEY"       # опц.: имя переменной для env-выдачи/авто-инъекта
sessions = ["*"]
confirm = false              # опц.: спрашивать кнопкой перед выдачей
```

Выдача (в сессии):

- `wallet get openai_dev` — печатает **значение** (вставить, ввести в форму);
- `wallet env openai_dev` — печатает `export OPENAI_API_KEY='…'`; модель делает
  `eval "$(wallet env openai_dev)"` или кладёт в `.env` сервиса;
- если у секрета задан `env`, переменная сразу доступна в окружении сессии с
  **реальным значением** (в отличие от inject, где там маркер) — сервис, который
  модель запускает, просто читает `OPENAI_API_KEY`. (Новый shared-секрет
  подхватится после resume/рестарта сессии.)

Границы: значение выдаёт ТОЛЬКО `shared` — host/inject значения не отдаются
никогда (эндпоинт `/get` их отклоняет). `commands`/`guard`/`deny` к shared не
применяются (команды нет), и обёртки в PATH под shared не создаются (значение уже
в env). Гейтится `sessions`/`confirm`, выдача видна строкой в статус-бабле.
**Только для дешёвых/лимитированных/dev-кредов** — модель значение видит, guard
его не страхует; боевые ключи — в деплой, не в shared.

## Настройка policy — справочник

**Принцип.** Кошелёк не должен душить модель (иначе проще прогать самому),
поэтому по умолчанию удобно: голое имя инструмента = любой его вызов. Но есть
пол, который держится всегда, — само условие смысла кошелька: модель должна
**использовать** креды, не **читая** их. Три слоя, в порядке применения:

1. **allow — `commands`** (где кошелёк доступен). Голое имя (`gh`, `git`, `ssh`) —
   любой вызов инструмента; строка с пробелом/глобом (`gh api repos/*`,
   `curl https://api/*`) — fnmatch по всей команде. Пусто у host-passthrough →
   дефолт `["gh","git","ssh","scp"]`; пусто у inject → ничего (сырой токен не
   открываем без явного списка). Из первого токена каждого шаблона строится
   обёртка в `.wallet-bin` (`curl https://…/*` → обёртка `curl`); чистые глобы
   (`*`) обёрткой не покрываются — для них только ручной `wallet run/exec`.
2. **guard — встроенный всегда-запрет** (поверх allow). Рубит опасное независимо
   от `commands`, с **прозрачным объяснением модели** (что не так и как правильно
   — доходит до её терминала): команды, печатающие сам токен (`gh auth token`,
   `gh auth status --show-token`), и git-RCE через флаги/транспорт (`git -c …`,
   `ext::…`, `--receive-pack`/`--upload-pack`/`--exec`). Отключается глобально
   `WALLET_GUARD=0` или точечно на секрет `allow_unsafe = true`.
3. **deny — точечный запрет секрета** (поверх allow, deny побеждает). Голый токен
   (`--force`, `--hard`) блокирует этот флаг/аргумент где угодно; строка с
   пробелом/глобом — fnmatch по всей команде. Для «разрешаю инструмент, но не эти
   опасные опции».

**Поля записи `[secrets.<имя>]`:**

| поле | смысл | дефолт |
|---|---|---|
| `value` + `env` | inject: значение и env-переменная для него | нет = host-passthrough |
| `description` | человекочитаемое; помогает модели выбрать нужный секрет | пусто |
| `sessions` | fnmatch имён сессий, кому доступен (`*` = все) | нет = никому |
| `commands` | allow-лист (голые имена/шаблоны) | host: `gh git ssh scp`; inject: ничего |
| `deny` | точечный запрет поверх allow | нет |
| `allow_unsafe` | снять встроенный guard на этом секрете | `false` |
| `confirm` | кнопки ✅/❌ перед запуском (и перед выдачей shared) | `true` |
| `shared` | значение ВЫДаётся сессии (`wallet get`/`env`); см. «Режим shared» | `false` |

**Что по умолчанию** (кошелёк включён + запись без лишних полей):
host-passthrough с `gh/git/ssh/scp`, guard включён, `confirm=true`. Хватает для
«дай сессии работать с git/gh хоста, не пуская токен в модель».

**Примеры:**

```toml
# Хостовые креды для всех сессий (keyring/gh/git/ssh) — удобно и безопасно:
[secrets.host]
sessions = ["*"]
commands = ["gh", "git", "ssh", "scp"]   # любой вызов; guard рубит опасное

# Разрешить инструмент, но запретить разрушительные флаги:
[secrets.careful-git]
sessions = ["ci-*"]
commands = ["git"]
deny = ["--force", "--force-with-lease", "git push --delete*"]

# Тонко: только чтение через gh api:
[secrets.readonly]
sessions = ["report"]
commands = ["gh api user", "gh api repos/*"]

# Доверенный случай, где нужен как раз git -c (guard мешает):
[secrets.special]
sessions = ["build"]
commands = ["git"]
allow_unsafe = true    # ОСОЗНАННО снимаем guard на этом секрете
confirm = true
```

## Правка policy из бота (`/wallet`)

Policy можно смотреть и править из Telegram, не открывая файл на хосте:

- `/wallet` — список секретов и policy в код-блоке (**значения токенов не
  показываются**) + подсказка по правке прямо в ответе;
- `/wallet confirm <имя> on|off` — спрашивать ли кнопкой перед запуском;
- `/wallet session <имя> +<шаблон>|-<шаблон>` — кому доступен (`*`, `dev-*`);
- `/wallet cmd <имя> +<шаблон>|-<шаблон>` — какие команды (`gh`, `git`, `gh api *`);
- `/wallet deny <имя> +<шаблон>|-<шаблон>` — точечный запрет поверх commands;
- `/wallet new <имя>` — создать host-passthrough секрет (пустой, deny-by-default);
- `/wallet rm <имя>` — удалить секрет.

Правки пишутся в `secrets.toml` атомарно (tmp+`os.replace`, права 0600
сохраняются, комментарии не теряются — `tomlkit`) и подхватываются демоном по
mtime, **без рестарта**. Обёртки новой сессии пересобираются под актуальный
`commands`; уже запущенной — при её следующем provision (resume/рестарт).

Границы: **значения токенов** (inject `value`/`env`) и `allow_unsafe` через бота
не трогаются — только правкой host-файла (иначе токен утёк бы в историю чата).
Команда только для `ALLOWED_USER_IDS`; модель вызвать её не может (приходит от
пользователя, не из Claude). Глобальный тумблер **`WALLET_POLICY_EDIT`** (по
умолчанию включён; `0` — только просмотр, правки лишь host-файлом).

## Слои защиты — единая связка «песочница + кошелёк»

Конфиденциальность держится не на одном барьере, а на слоях, которые **не
работают порознь**:

1. **Песочница (`SANDBOX=bwrap`) — фундамент.** Прячет от сессии сырые креды
   хоста. Без неё модель читает всё напрямую — остальное теряет смысл.
2. **Кошелёк — контролируемый доступ обратно.** Даёт применять креды, не читая
   их. Барьеры: `sessions`/`commands`/`deny`/`confirm` + встроенный guard.
3. **Судья auto-режима + `permissions.deny` — блок попыток в обход.** В
   `PERMISSION_MODE=auto` действия сессии оценивает отдельный классификатор
   Claude Code (в 2.1.2x по умолчанию Sonnet 5); его промпт не кастомизируется, а
   правила — да, оркестратор пишет их в `settings.local.json` сессии:
   - `autoMode.hard_deny` («не добывай секреты; для git/gh — кошелёк»),
     `autoMode.environment` (где лежат секреты), `classifyAllShell` (проверять
     ВСЕ bash-команды; из `AUTOMODE_CLASSIFY_ALL_SHELL`, дефолт on);
   - `permissions.deny` — жёсткое чтение кред-файлов (`~/.config/gh`, `~/.netrc`,
     `~/.git-credentials`, `~/.ssh`, `~/.aws`, keyring-CLI). Работает во ВСЕХ
     режимах, включая `bypass`.

**Ключевое:** слой 3 добавляется ТОЛЬКО когда активна связка — кошелёк включён
**и** `SANDBOX=bwrap`. Без песочницы правила были бы театром, без кошелька нечего
защищать. Всё это включается только целиком.

**Защита — механика, не уговоры в промпте.** Под bwrap хостовые креды физически
недостижимы для модели, а единственная точка, где значение секрета течёт к ней —
вывод команды под кошельком — редактируется в демоне (значение → `•••`). Обёртки и
подсказка в промпте лишь **дают пользоваться прозрачно** (зови инструмент как
обычно, `$NAME` для токена, «не смог — скажи оператору») — они не барьер.

## Модель угроз и известные ограничения

Bwrap прячет `~/.ssh`, `~/.aws` и прочий дом, но всё, что мы *дали бы* сессии для
работы CLI-тулзы (env, `~/.netrc`, файл конфига), модель по определению может
прочитать. Значит секрет не должен быть в адресном пространстве песочницы вообще
— только результат его применения. Отсюда — остаточные дыры, которые
закрываются **доверием (policy), а не аргументами**:

- **Команда исполняется в cwd проекта, куда модель пишет.** `git push` (через
  обёртку) запускает git на хосте в папке проекта; модель может подложить туда
  `./.git/config` (`core.sshCommand`/hooks) и получить исполнение на хосте с
  секретом в env. Узкий шаблон команды это НЕ закрывает; для inject демон
  выставляет `GIT_CONFIG_NOSYSTEM`/`GIT_CONFIG_GLOBAL=/dev/null` (частично).
  **Настоящий барьер — `sessions` (доверенные) + `confirm` (человек видит
  команду) + host-passthrough (значения нет).**
- **Обёртка гонит инструмент на хост целиком.** Завёрнутый `curl`/`gh`
  исполняется на хосте, а не в песочнице; поэтому `commands` должны быть узкими
  (URL/подкоманды), а не `*`. Обёртки создаются только для инструментов из
  `commands` (ownership-fork: чего не дал — того не завернём).
- **Команды под кошельком — строго неинтерактивные.** Демон запускает их без
  TTY и гасит GUI-askpass (`SSH_ASKPASS_REQUIRE=never`, `GIT_TERMINAL_PROMPT=0`,
  снят `DISPLAY`), поэтому интерактивные запросы (host-key незнакомого ssh-хоста,
  git-credential-промпт) **падают с ошибкой в stderr**, а не висят GUI-диалогом
  на хосте. Модель должна звать инструменты неинтерактивно (`ssh -o BatchMode=yes
  -o StrictHostKeyChecking=accept-new`, git с флагами). Отдельно: сама песочница
  по умолчанию вырезает `DISPLAY`/X (`SANDBOX_X11`, см. README), так что и
  незавёрнутый процесс не дотянется до хостового GUI.
- **Редакция вывода — literal-only, best-effort.** Заменяются точные вхождения
  значений; любая трансформация в команде (base64, reverse, запись в файл)
  обходит. Это гигиена против случайного эха, не барьер.
- **Шаблоны команд — fnmatch по `" ".join(cmd)`.** `*` пересекает границы argv и
  пробелы. Пиши шаблоны консервативно (префикс + явные аргументы), не полагайся
  на `*` для ограничения хвоста.
- **Поддержан только `SANDBOX=bwrap`** (см. «Зачем песочница»).

Значения секретов (inject/shared) вымарываются не только из вывода команд, но и
из **исходящего в чат текста** — reply модели, служебных уведомлений и строк
статус-бабла (`OrchestratorCore._scrub` через редакторы модулей). Safety-net:
случайный `echo` shared-значения или запись ключа в `.env` не улетят в Telegram/
историю. Не барьер (модель значение держит), гигиена.

## Ручной путь и справка

Обёртки — норма, но CLI `wallet` остаётся для отладки и явного управления:

- `wallet ls` — что доступно (имя, режим, разрешённые команды);
- `wallet run <имя> -- <cmd>` — выполнить с КОНКРЕТНЫМ секретом (по имени);
- `wallet exec <cmd>` — выполнить, секрет подберётся по команде (это и зовут
  обёртки);
- `wallet get <имя>` / `wallet env <имя>` — значение shared-секрета;
- `wallet help` — полная справка (та же, что печатает CLI).

## Что отложено

- **MCP-тул `wallet_run`** (то же, что CLI, но нативно в permission-механизме
  Claude Code) — не реализован; интерфейс — обёртки в PATH + CLI `wallet`.
- **TLS-прокси с подстановкой токена** (подмена в исходящем запросе, чтобы
  значение вообще не попадало на хост-команду) сильнее по конфиденциальности, но
  отложен: TLS-MITM хрупок (cert-pinning ломает его в ноль), покрывает только
  HTTP-секреты (не SSH-ключи/сертификаты), и дёшев только на microVM.
  **Расчёт «возьмём прокси agent-vm» НЕ оправдался** (проверено 2026-07-21): его
  прокси захардкожен под три источника — `~/.claude/.credentials.json`,
  `~/.codex/auth.json`, `gh auth token`; конфиг-файла и флагов для произвольных
  хостов/секретов нет, а `<hash>.secrets` в его state-каталоге — собственное
  хранилище, вход туда не предусмотрен. Так что переиспользовать его под секреты
  кошелька нельзя; свой TLS-слой остаётся отдельной, никем не начатой работой.
  Подробности и решение по кошельку в VM-режиме —
  [agent-vm-integration.md](agent-vm-integration.md).
